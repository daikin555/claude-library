## 原文（日本語訳）

非ASCII文字を含むBashコマンドの解析が、実際のシェルの単語境界と一致するように修正しました。

## 原文（英語）

Fixed Bash command parsing of non-ASCII characters to match real shell word boundaries

## 概要

Claude Codeの権限ルール（`Bash(prefix *)` のようなワイルドカードパターン）は、コマンド文字列を「単語」単位に分割し、その境界をもとにマッチ判定を行っています。これまでこの単語分割ロジックが、日本語・中国語・絵文字・アクセント付き文字などの非ASCII（マルチバイト）文字を含むコマンドに対して、実際のbashの単語分割（word splitting）とは異なる境界を検出してしまうことがありました。今回の修正により、非ASCII文字を含むコマンドでも実際のシェルと同じ単語境界でパースされるようになり、権限ルールのマッチ判定がより正確になりました。

## 基本的な使い方

これは内部的なパーサーの修正であり、ユーザー側の設定変更は不要です。Claude Codeをv2.1.216以降にアップデートするだけで、以下のような挙動が自動的に改善されます。

- 非ASCII文字を含むファイルパスやコマンド引数を持つBashコマンドが、権限ルールと正しくマッチするようになる
- `Bash(prefix *)` のような末尾ワイルドカード付き許可/拒否ルールが、非ASCII文字の直後でも意図通りの単語境界で判定される

## 実践例

### 日本語のファイル名・パスを含むコマンドの許可ルール

`.claude/settings.json` で日本語を含むディレクトリに対するコマンドを許可する場合、以前は単語境界の誤判定によって意図しないマッチ／非マッチが起きる可能性がありましたが、修正後は実際のシェルと同じ境界でマッチします。

```json
{
  "permissions": {
    "allow": [
      "Bash(cat ドキュメント/*)",
      "Bash(ls 資料フォルダ/*)"
    ]
  }
}
```

### 絵文字やアクセント付き文字を含む引数

コミットメッセージや検索クエリに絵文字・アクセント付き文字（例: `café`, `🚀`）が含まれる場合でも、実際のシェルのトークナイズ結果と一致した単語境界でパターンマッチが行われます。

```bash
git commit -m "リリース準備完了 🚀"
```

### 既存の許可ルールの動作確認

アップデート後、非ASCII文字を含むコマンドで権限プロンプトが想定通りに出るか（あるいは出ないか）を確認したい場合は、`/permissions` コマンドで現在の許可・拒否ルールを確認できます。

```
/permissions
```

## 注意点

- この修正はバグ修正であり、既存の `settings.json` の権限ルールを書き換える必要はありません
- 非ASCII文字を含むコマンドに対して、これまで意図せず許可・拒否されていたケースが変わる可能性があるため、CI等で厳密な権限ルールを運用している場合はアップデート後に一度動作確認することを推奨します
- ワイルドカードの単語境界の基本的な考え方（`Bash(ls *)` は `ls -la` にマッチするが `lsof` にはマッチしない、など）自体は変わっていません
