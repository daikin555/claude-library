## 原文（日本語訳）

`EnterWorktree` は、プロジェクトの `.claude/worktrees/` ディレクトリ以外にあるgit worktreeに入る前に確認を求めるようになった。

## 原文（英語）

`EnterWorktree` now asks for confirmation before entering a git worktree outside the project's `.claude/worktrees/` directory

## 概要

Claude Codeは `EnterWorktree` ツールを使って、セッション中に別のgit worktreeへ切り替えることができます。これまでは対象パスがどこにあっても確認なしで切り替えが行われていましたが、今回のアップデートにより、切り替え先がプロジェクトの `.claude/worktrees/` ディレクトリの外にある場合には、実行前にユーザーへ確認を求めるようになりました。手動で作成したworktreeやカスタムフックで別の場所に作られたworktreeへ、意図せずClaudeの作業ディレクトリが切り替わってしまうことを防ぐための安全対策です。

## 基本的な使い方

通常、`.claude/worktrees/` 配下のworktree（`--worktree` フラグやセッション中に「worktreeで作業して」と依頼した場合に作られるもの）へClaudeが切り替える際は、これまで通り確認なしで動作します。

```bash
claude --worktree feature-auth
```

セッション中にClaudeへ次のように依頼すると、内部的に `EnterWorktree` ツールが呼び出されます。

```text
worktreeで作業して
```

## 実践例

### 手動で作成した、プロジェクト外のworktreeに入らせる場合

`git worktree add` でリポジトリの外側にworktreeを手動作成し、Claudeにそこで作業させたいケースです。

```bash
git worktree add ../project-feature-a -b feature-a
```

このパス（`.claude/worktrees/` の外）へClaudeが `EnterWorktree` で切り替えようとすると、今回のアップデートにより確認ダイアログが表示されます。意図した操作であれば承認し、身に覚えがない場合は拒否できます。

### `WorktreeCreate` フックでプロジェクト外にworktreeを作成している場合

SVNなど非gitのバージョン管理向けに `WorktreeCreate` フックを使い、`$HOME/.claude/worktrees/$NAME` のようにプロジェクトルートの `.claude/worktrees/` 以外の場所へworktreeを作成する設定をしているケースです。この場合もClaudeが該当worktreeに入ろうとすると確認が求められるため、フック側の設定意図とClaude側の挙動を突き合わせて確認できます。

### CI・非対話環境での挙動を把握しておく

`-p` を使った非対話実行では、確認プロンプトに応答できないため、プロジェクト外worktreeへの `EnterWorktree` が必要な自動化フローを組んでいる場合は、事前に対象worktreeを `.claude/worktrees/` 配下に配置するなど設計を見直しておくとよいでしょう。

## 注意点

- 確認が必要になるのは、あくまで対象が「プロジェクトの `.claude/worktrees/` ディレクトリの外」にある場合のみです。標準的な `--worktree` フローで作られたworktree間の切り替えでは、これまで通り確認は挟まりません。
- この変更はセキュリティ・安全性の観点からの改善であり、Claudeが意図せず信頼できない場所やプロジェクト外のディレクトリで作業してしまうリスクを減らします。
