## 原文（日本語訳）

ネットワークの外部通信制御は維持したまま、ファイルシステムの分離だけをスキップできる `sandbox.filesystem.disabled` 設定を追加しました。

## 原文（英語）

Added `sandbox.filesystem.disabled` setting to skip filesystem isolation while keeping network egress control

## 概要

Claude Codeのサンドボックス機能には、ファイルシステムアクセスの分離とネットワーク送信（egress）の制御という2つの独立した保護レイヤーがあります。今回追加された `sandbox.filesystem.disabled` 設定を使うと、ファイルシステム分離だけを無効化しつつ、ネットワークの外部通信制御はそのまま有効な状態を維持できます。プロジェクト構成の都合でファイルシステムの分離が動作しにくい環境（マウント構成が複雑な場合や、複数ディレクトリへの書き込みが頻繁に必要な場合など）でも、ネットワーク面のガードレールだけは失わずに済みます。

## 基本的な使い方

`.claude/settings.json` にサンドボックス設定を追加し、ファイルシステム分離のみを無効化します。

```json
{
  "sandbox": {
    "filesystem": {
      "disabled": true
    }
  }
}
```

この設定を有効にすると、Bashツールなどのサンドボックス実行時にファイルシステムへの読み書き制限（`denyRead` / `denyWrite` によるOSレベルの分離）が適用されなくなります。一方でネットワークの許可リスト（egressのallow/deny設定）はそのまま機能します。

## 実践例

### ネットワーク制御だけを維持したい開発環境

外部への通信は制限しつつ、ローカルファイルシステムへの複雑な書き込みパターン（シンボリックリンク越しのビルド成果物出力など）を通したい場合の設定例です。

```json
{
  "sandbox": {
    "filesystem": {
      "disabled": true
    },
    "network": {
      "allowedDomains": ["registry.npmjs.org", "github.com"]
    }
  }
}
```

### CI環境でのマウント構成との相性問題を回避

Dockerコンテナ内やネストしたサンドボックス環境では、ファイルシステム分離のためのマウント設定が既存のCI構成と衝突することがあります。そのような場合にファイルシステム分離だけを切り離し、ネットワークegress制御はCIのセキュリティ要件として維持します。

```json
{
  "sandbox": {
    "filesystem": {
      "disabled": true
    },
    "allowUnsandboxedCommands": false
  }
}
```

### 既存のサンドボックス設定との組み合わせ確認

`/sandbox` コマンドや `dangerouslyDisableSandbox` オプションと組み合わせる場合、ファイルシステム側だけを無効化した状態でどのツール呼び出しが分離対象になっているかを確認しておくと安全です。

```bash
# 現在のサンドボックス設定を確認
claude
> /sandbox
```

## 注意点

- この設定はファイルシステムの分離のみを無効化するものであり、ネットワークの外部通信制御には影響しません
- ファイルシステム分離を無効化すると、プロジェクトディレクトリ外への意図しない書き込みを防ぐOSレベルの保護がなくなるため、信頼できる環境でのみ使用してください
- チーム全体に適用する場合は、`.claude/settings.json`（プロジェクト共有設定）ではなく個人のローカル設定で有効化するかどうかを慎重に検討してください
- 既存の `allowUnsandboxedCommands` や `excludedCommands` などのサンドボックス関連設定と合わせて動作を確認することを推奨します
