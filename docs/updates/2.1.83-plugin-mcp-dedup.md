## 原文（日本語に翻訳）

組織管理コネクターと重複するプラグイン MCP サーバーは、2 つ目の接続を実行するのではなく抑制されるようになりました。

## 原文（英語）

Plugin MCP servers that duplicate an org-managed connector are now suppressed instead of running a second connection

## 概要

組織の IT 管理者が設定した MCP コネクターと同じサーバーをプラグインも提供しようとした場合、以前は 2 つの接続が確立されていました（重複接続）。本改善により、プラグインの重複 MCP サーバーは自動的に抑制され、組織管理のコネクターのみが使用されます。これにより、重複接続によるリソース無駄遣いや予期しない動作が防止されます。

## 基本的な使い方

この改善は自動的に適用されます。ユーザーの操作変更は不要です。

```bash
# 組織管理コネクター: slack-managed
# プラグインが提供する同じ Slack コネクター: slack-plugin
# 修正後: slack-plugin が自動的に抑制され slack-managed のみが使用される
```

## 実践例

### ユースケース: 企業環境での Slack コネクター管理

組織が Slack MCP コネクターを管理している場合、プラグインの重複コネクターが抑制されます。

```json
// 組織の managed-settings.json（管理者が設定）
{
  "mcpServers": {
    "slack-org": {
      "url": "https://mcp.example.com/slack"
    }
  }
}
```

```bash
# ユーザーがインストールしたプラグインも Slack コネクターを持っている場合
# 修正後: 組織のコネクターが優先され、プラグインのコネクターは抑制される
$ claude
# Slack への接続は 1 つのみ（組織管理）
```

### ユースケース: ポリシーの一貫性確保

組織のセキュリティポリシーに従ったコネクターのみが使用されることを確保できます。

```bash
# 管理者視点: プラグインが勝手に追加の MCP 接続を作らない
# ユーザー視点: 意図せず重複接続が発生しない
```

## 注意点

- 重複の判断は MCP サーバーの URL やサーバー名に基づいて行われます
- プラグインのコネクターが「抑制」されても、エラーにはなりません（ログに記録されます）
- この機能は組織管理コネクターが存在する場合のみ効果があります
- 個人環境（組織管理なし）では引き続きすべてのプラグイン MCP サーバーが有効です
