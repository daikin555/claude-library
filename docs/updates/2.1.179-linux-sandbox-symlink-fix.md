## 原文（日本語に翻訳）

`.claude/skills`または`.claude/hooks`がシンボリックリンクの場合にLinuxサンドボックスが起動しない問題を修正

## 原文（英語）

Fixed Linux sandbox failing to start when `.claude/skills` or `.claude/hooks` is a symlink

## 概要

Linuxのサンドボックス環境で、`.claude/skills`または`.claude/hooks`ディレクトリがシンボリックリンクとして設定されている場合、サンドボックスが起動に失敗する問題がありました。共有設定やdotfilesでスキルやフックを管理している開発者に影響していました。この修正により、シンボリックリンクを使用していてもLinuxサンドボックスが正常に起動します。

## 基本的な使い方

この修正は自動的に適用されます。シンボリックリンクで`.claude/skills`や`.claude/hooks`を管理している場合も、サンドボックスが正常に動作します。

```bash
# シンボリックリンクを使った設定（修正後は正常動作）
ln -s ~/dotfiles/claude/skills ~/.claude/skills
ln -s ~/dotfiles/claude/hooks ~/.claude/hooks

# Claude Code 起動（修正後：正常にサンドボックスが起動）
claude
```

## 実践例

### dotfilesで共有スキルを管理

```bash
# dotfilesリポジトリの構造
~/dotfiles/
├── claude/
│   ├── skills/
│   │   ├── deploy.md
│   │   └── review.md
│   └── hooks/
│       ├── pre-tool.sh
│       └── post-tool.sh

# シンボリックリンクを作成
ln -s ~/dotfiles/claude/skills ~/.claude/skills
ln -s ~/dotfiles/claude/hooks ~/.claude/hooks

# 修正後：正常にサンドボックスが起動
claude
# ✓ Skills loaded: deploy, review
# ✓ Hooks loaded: pre-tool, post-tool
```

### チーム共有設定のシンボリックリンク管理

```bash
# チームの共有設定をNFSやマウントポイントから参照
ln -s /shared/team/claude-skills ~/.claude/skills
ln -s /shared/team/claude-hooks ~/.claude/hooks

# 修正後：リモートマウントへのシンボリックリンクでも動作
claude
```

### Nixやhome-managerでの設定管理

NixOSやhome-managerを使用している場合、設定ファイルはストアへのシンボリックリンクになることがあります。

```nix
# home.nix
home.file.".claude/skills".source = ./claude/skills;
home.file.".claude/hooks".source = ./claude/hooks;
# → Nixストアへのシンボリックリンクが作成される
# 修正後：正常に動作
```

## 注意点

- この問題はLinux環境固有で、macOSでは発生しませんでした
- シンボリックリンクのターゲットが存在しない場合は引き続きエラーになります
- サンドボックスが起動しない場合は、シンボリックリンクのターゲットが正しく存在するか確認してください

```bash
# シンボリックリンクの確認方法
ls -la ~/.claude/skills
ls -la ~/.claude/hooks
# → lrwxrwxrwx ... -> /path/to/target のように表示される
# → ターゲットパスが実際に存在するか確認
```
