## 原文（日本語訳）

クラウドセッションで、ターンの途中にセッションのコンテナが再起動すると処理中のメッセージが失われる問題を修正しました。中断されたターンは、再開時に自動的に再実行されるようになり、セッションが応答不能なまま放置されることがなくなりました。

## 原文（英語）

Fixed cloud sessions dropping the in-flight message when the session's container restarts mid-turn — the interrupted turn now re-runs on resume instead of leaving the session unresponsive

## 概要

Claude Code on the web（クラウドセッション）は、Anthropicが管理するVM上でタスクを実行します。従来、Claudeが応答を生成している最中（ターンの途中）にセッションのコンテナが何らかの理由で再起動すると、処理中だったメッセージがそのまま失われ、セッションが応答しなくなってしまう問題がありました。今回の修正により、中断されたターンはセッション再開時に自動的に再実行されるようになり、ユーザーが手動で再送信しなくても処理が継続されるようになりました。

## 基本的な使い方

この修正は自動的に適用される信頼性改善のため、特別な操作は不要です。クラウドセッションでコンテナの再起動が発生した場合、セッションを再開すると中断されたターンが自動的に再実行されます。

```
# claude.ai/code でクラウドセッションを開始
claude --cloud "READMEを更新して"
```

万が一コンテナ再起動が発生しても、セッションを開き直せば処理が自動的に再開されます。

## 実践例

### 長時間実行タスク中のコンテナ再起動からの自動復旧

大規模なリファクタリングやテストスイートの実行など、長時間かかるタスクをクラウドセッションで実行している場合、インフラ側のメンテナンスなどでコンテナが再起動しても、再開時に中断していたターンから自動的にやり直されます。

```bash
# クラウドセッションでの長時間タスクの例
claude --cloud "全テストスイートを実行して失敗したテストを修正して"
```

### モバイルアプリからセッションを監視している場合

Claudeモバイルアプリでクラウドセッションの進捗を監視している際に、セッションが「応答なし」の状態で止まっているように見えても、この修正以降は再開操作によって中断されたターンが再実行されるため、セッションを最初からやり直す必要がなくなります。

### `--teleport` でローカルに引き継ぐ前の状態確認

クラウドセッションをローカル環境に `--teleport` で引き継ぐ前に、コンテナ再起動があったセッションでもメッセージ履歴が欠落していないことを確認できます。

```bash
claude --teleport <session-id>
```

## 注意点

- 再実行されるのはコンテナ再起動によって中断された「ターン」であり、すでに正常に完了しているターンの内容が変更されることはありません。
- コンテナ再起動の頻度自体を減らす設定ではなく、あくまで再起動が発生した場合の復旧挙動を改善するものです。
- ターンの再実行によって、同じ処理が重複して実行される可能性があるツール呼び出し（副作用のあるAPI呼び出しなど）が含まれる場合は、再実行後の結果を確認することをおすすめします。
