## 原文（日本語訳）

OAuth認証を使うMCPサーバーが、トークンリフレッシュに1回失敗しただけで手動の再認証を要求してしまう不具合を修正した。

## 原文（英語）

Fixed OAuth MCP servers requiring manual re-authentication after a single failed token refresh

## 概要

OAuthで認証するMCPサーバー（NotionやAtlassianなどの外部サービス連携）では、アクセストークンの有効期限が切れるとリフレッシュトークンを使って自動的に再取得する仕組みになっています。これまでは、このトークンリフレッシュ処理が一時的なネットワークエラーなどで1回失敗しただけで、すぐに手動の再認証（`/mcp` からの再ログイン）が必要になってしまう不具合がありました。今回の修正により、単発のリフレッシュ失敗だけでは即座に手動再認証を要求しないようになり、一時的な失敗に対する耐性が向上しています。

## 基本的な使い方

OAuth対応のMCPサーバーを通常どおり設定し、初回のみ `/mcp` から認証を行います。

```text
/mcp
```

以降のセッションでは、トークンのリフレッシュが自動的に行われるため、通常は再認証操作は不要です。

## 実践例

### 長時間セッションを継続する場合

長時間Claude Codeのセッションを開いたままにしていると、途中でOAuthアクセストークンの有効期限が切れることがあります。以前は最初のリフレッシュ試行が何らかの理由で失敗すると即座に再認証を求められていましたが、修正後は一時的な失敗であれば自動的にリトライされ、作業を中断されにくくなります。

```text
（作業を継続中に自動でトークンがリフレッシュされる）
```

### ネットワークが不安定な環境でのMCP利用

Wi-Fiの瞬断やVPN切り替えなど、ネットワークが一時的に不安定になる環境でOAuth連携のMCPサーバーを使う場合も、単発のリフレッシュ失敗によって毎回再認証を求められるという煩わしさが軽減されます。それでもリフレッシュ自体が継続的に失敗する場合は、`/mcp` から手動で再認証してください。

```text
/mcp
```

## 注意点

- リフレッシュトークン自体が失効・無効化されている場合など、根本的に再認証が必要なケースでは、引き続き `/mcp` からの手動再認証が必要です。
- 本修正は「1回の失敗で即座に再認証を要求する」挙動を改善するものであり、OAuthフロー自体の仕様（アクセストークンやリフレッシュトークンの有効期限など）が変更されたわけではありません。
- MCPサーバー側のOAuth実装によっては、依然としてトークンの永続化や自動リフレッシュに関する制約が残っている場合があります。
