## 原文（日本語訳）

`CLAUDE_CODE_PROCESS_WRAPPER` を追加：Agent Viewとバックグラウンドサービスが企業のランチャーに対応し、Claude Codeが自身を再起動・追加起動する際は必ず指定のラッパー実行ファイルを経由するようになった。

## 原文（英語）

Added `CLAUDE_CODE_PROCESS_WRAPPER`: agent view and the background service now honor a corporate launcher by running every Claude Code self-spawn through a required wrapper executable

## 概要

企業環境では、認証情報の注入やセキュリティポリシーの適用のために、`claude` コマンド自体を独自のラッパースクリプト経由で起動することを要求される場合があります。これまで `claude agents`（Agent View）やバックグラウンドサービスがセッション管理のために内部的にClaude Codeの新しいプロセスを自己起動する際、このラッパーを経由せず直接 `claude` 実行ファイルを呼び出してしまうケースがありました。`CLAUDE_CODE_PROCESS_WRAPPER` はこの挙動を変更し、Claude Codeが自身を再起動・追加起動するすべてのケースで指定したラッパー実行ファイルを必ず経由させる環境変数です。

## 基本的な使い方

環境変数に、Claude Codeの起動をラップする実行ファイルのパスを指定します。

```bash
export CLAUDE_CODE_PROCESS_WRAPPER=/usr/local/bin/corp-claude-launcher
claude agents
```

これにより、Agent Viewがバックグラウンドセッションを起動する際にも、指定したラッパー経由で `claude` プロセスが起動されるようになります。

## 実践例

### 社内認証スクリプトを経由してバックグラウンドセッションを起動する

社内のシークレット管理システムからAPIキーを取得するラッパースクリプトを用意し、そのスクリプトを常に通過させたい場合に設定します。

```bash
#!/bin/bash
# /usr/local/bin/corp-claude-launcher
source /etc/corp/inject-secrets.sh
exec claude "$@"
```

```bash
export CLAUDE_CODE_PROCESS_WRAPPER=/usr/local/bin/corp-claude-launcher
```

この状態で `claude agents` からバックグラウンドセッションを複数ディスパッチしても、それぞれのセッションが同じラッパーを経由して起動されるため、認証情報の注入漏れを防げます。

### CI/管理された端末でのポリシー強制

社給端末の管理ポリシーとして、Claude Codeの起動時に必ず監査ログを記録するラッパーを通す運用をしている組織では、この環境変数をmanaged settingsやMDM経由で全社的に配布することで、Agent Viewやバックグラウンドサービスも含めたすべての自己起動プロセスにポリシーを適用できます。

## 注意点

- 本記事作成時点では、`CLAUDE_CODE_PROCESS_WRAPPER` は公式の環境変数リファレンスページにはまだ掲載が確認できませんでした。changelogの記載内容をもとに記述しているため、正確な仕様（ラッパーに渡される引数の形式など）は今後のドキュメント更新を確認してください。
- 主にAgent View（`claude agents`）とバックグラウンドサービスにおける自己起動（self-spawn）が対象であり、通常のインタラクティブセッションの初回起動そのものに影響するかどうかは明記されていません。
- VS Code拡張機能には類似の設定として `claudeCode.claudeProcessWrapper` がありますが、これは別の設定項目であり、本環境変数とは異なるものです。
