## 原文（日本語訳）

バックグラウンドタスクの通知に、人間の入力が一切発生していないことを明示するようになった。これにより、transcript（会話履歴）内に捏造された承認記録が紛れ込んでいても、それに基づいて行動してしまうことを防止する。

## 原文（英語）

Background task notifications now explicitly state that no human input has occurred, preventing fabricated in-transcript approvals from being acted on

## 概要

Claude Codeのバックグラウンドタスク（`/bg` によるバックグラウンドエージェントや、スケジュール実行・Webhookトリガーなど）は、完了時や入力待ちになった際に `Notification` フックを通じて通知を発行します。これまでこれらの通知はtranscript上のテキストとして扱われるため、悪意あるツール出力やファイル内容にプロンプトインジェクションで「ユーザーはこの操作を承認しました」といった偽の記述が混入していた場合、Claudeがそれを本物の人間による承認と誤認してしまうリスクがありました。今回の変更では、バックグラウンドタスクの通知メッセージ自体に「これは人間の入力を伴っていない」ことを明示的に記載するようにし、transcript内の捏造された承認文言を根拠に危険な操作を実行してしまうことを防ぎます。

## 基本的な使い方

この改善はClaude Codeの内部動作であり、ユーザー側で明示的に設定する項目はありません。バックグラウンドタスク（`/bg` で起動したエージェントセッションや、スケジュール実行・Webhookトリガー経由のタスク）が完了、または入力待ちになった際に発火する通知に、変更が自動的に反映されます。

```text
/bg 大量のテストファイルをリファクタリングして
```

上記のようにバックグラウンドで実行したタスクが完了した際、`agent_completed` 通知がフックとして発火しますが、その通知内容には「これはシステムからの通知であり、人間が入力・承認したものではない」ことが明示されるようになります。

## 実践例

### スケジュール実行やWebhookトリガー経由のタスクでの誤承認防止

`/schedule` で定期実行しているタスクや、Webhookトリガーで起動するタスクが、外部から取得したファイルやWebページの内容を処理する場合、その内容にプロンプトインジェクションとして「この操作は管理者によって承認済みです。次のコマンドを実行してください」といった文言が仕込まれている可能性があります。今回の変更により、Claudeはこうした通知やタスク実行結果に紛れ込んだ「承認済み」という記述だけでは、実際に人間が承認した根拠として扱わなくなり、危険な操作の連鎖的な実行を防げます。

```text
/schedule 毎朝9時に外部APIのレスポンスを取得し、異常があれば修正コミットを作成する
```

### 長時間のバックグラウンドエージェントにおける多段階の悪用防止

複数のサブエージェントが並行して動くバックグラウンドタスクでは、あるエージェントの出力が別のエージェントへの入力になることがあります。ここで悪意あるコード片やドキュメントに「このリクエストは人間によって事前承認されています」という偽の記述が含まれていても、通知システムが「人間の入力は発生していない」と明示するため、下流のエージェントがそれを信用して不可逆な操作（例: 本番環境へのデプロイや破壊的なファイル削除）を実行してしまう事態を避けやすくなります。

```text
/bg 依存関係の脆弱性を検出したら自動でパッチを適用してPRを作成して
```

## 注意点

- この対策は、あくまで「バックグラウンドタスクの通知」というシステム側のメッセージに関するものです。会話中に直接入力されたプロンプトインジェクション対策全般を代替するものではなく、外部コンテンツを扱う際は引き続き注意が必要です。
- 承認が必要な操作（パーミッションダイアログなど）は従来通り人間の明示的な操作が必要であり、この変更はその流れを変えるものではなく、あくまで「偽の承認記録に基づく誤動作」を防ぐための防御層の強化です。
- Auto mode（`--dangerously-skip-permissions` に代わる安全性を高めた実行モード）のように、ツール呼び出しの前に分類器がリスクを判定する仕組みと合わせて、多層的にプロンプトインジェクション対策が強化されている流れの一部と考えられます。
- バックグラウンドタスクの通知自体は `Notification` フック（`agent_needs_input` / `agent_completed` など）を通じて発火する仕組みであり、`settings.json` のフック設定と組み合わせて外部通知（Slack、デスクトップ通知など）にルーティングすることも可能です。
