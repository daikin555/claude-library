## 原文（日本語訳）

`/usage` が新しいデータより古いキャッシュ済みのバーを表示してしまう不具合と、設定編集後に `/mcp` がプレースホルダーサーバーを再分類できない不具合を修正した。

## 原文（英語）

Fixed `/usage` showing stale cached bars over fresher data, and `/mcp` not reclassifying placeholder servers after config edits

## 概要

`/usage` コマンドは、Pro・Max・Team・Enterpriseプランの利用状況をバーグラフで表示しますが、このデータはローカルにキャッシュされ、直近に取得した値を再利用する仕組みになっています。これまでは、より新しいデータが取得できているにもかかわらず、古いキャッシュ済みのバーが優先して表示されてしまう不具合がありました。また `/mcp` コマンドは接続されているMCPサーバーの状態を一覧表示しますが、設定ファイル（`.mcp.json` など）を編集してサーバー構成を変更した後も、以前「プレースホルダー」として扱われていたサーバーの分類が更新されず、実際の状態を反映しないままになる不具合もありました。今回のアップデートで、両方とも最新の状態が正しく反映されるようになりました。

## 基本的な使い方

利用状況を確認するには、セッション内で次のコマンドを実行します。

```text
/usage
```

MCPサーバーの状態を確認・再接続するには次のコマンドを使います。

```text
/mcp
```

## 実践例

### 長時間セッションでの利用状況チェック

長時間の作業中に定期的に `/usage` を確認して、レート制限にどれだけ近づいているかを把握したい場合があります。

```text
/usage
```

修正前は、直近で取得できたはずの新しい利用状況ではなく、それより前のキャッシュされたバーが表示されてしまうことがありましたが、修正後は常に最新のデータが優先して表示されます。利用状況の取得自体がレート制限された場合は、直近60分以内に取得できていたバーを「Showing last-known usage」という注記付きで表示し、`r` キーで再取得できます。

### MCP設定を編集した直後にサーバー状態を確認する

`.mcp.json` を編集してサーバーの接続情報を修正したあと、`/mcp` で状態を確認するケースです。

```json
// .mcp.json を編集してサーバー設定を修正
{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://example.com/mcp"
    }
  }
}
```

```text
/mcp
```

修正前は、設定を変更したにもかかわらず、以前プレースホルダー（未設定・認証待ちなど）扱いだったサーバーの分類がそのまま残ってしまうことがありましたが、修正後は設定編集後の状態が正しく反映され、`reconnect` や `enable`/`disable` サブコマンドで扱う対象も最新の構成に基づいたものになります。

## 注意点

- `/usage` のバーはこのマシン上のローカルなセッション履歴から計算された概算値であり、他のデバイスやclaude.aiでの利用は含まれません。
- `/mcp` はサーバー名や `all` を指定して `reconnect`・`enable`・`disable` を実行できます。ヘッドレスモード（`-p`）でも引数なしで実行すればテキストでサマリーを取得できます（v2.1.205以降）。
- どちらの不具合も表示上の問題であり、実際の利用量制限やMCPサーバーの接続そのものの挙動を変えるものではありません。
