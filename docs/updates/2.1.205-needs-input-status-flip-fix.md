## 原文（日本語訳）

エージェントのターンに読み取り可能なテキストが含まれていなかった場合に、バックグラウンドジョブがエージェント一覧で「入力待ち（needs input）」から「作業中（working）」に誤って戻ってしまう不具合を修正。

## 原文（英語）

Fixed background jobs flipping from "needs input" back to "working" in the agent list when the agent's turn contained no readable text

## 概要

`claude agents` で開くエージェント一覧（agent view）は、各バックグラウンドセッションの状態を「Working」「Needs input」「Idle」「Completed」「Failed」「Stopped」といった状態アイコンとHaikuクラスのモデルが生成する一行サマリーで表示します。今回の修正前は、Claudeが質問を投げかけて「Needs input」状態になった後、そのターンの応答にツール呼び出しのみが含まれ人間が読めるテキストが存在しない場合に、状態表示が誤って「Working」に戻ってしまうことがありました。実際にはセッションはユーザーの返答を待ったまま停止しているにもかかわらず、一覧上は作業中と表示されるため、対応が必要なセッションを見落とす原因になっていました。

## 基本的な使い方

エージェント一覧を開いて、複数のバックグラウンドセッションの状態をまとめて確認します。

```bash
claude agents
```

「Needs input」セクションに表示された行は、Claudeが具体的な質問や権限確認への回答を待っている状態です。`Space` キーでpeekパネルを開けば、待たれている内容を確認しながら返信できます。

## 実践例

### 複数のバックグラウンドタスクを並行監視する

複数のリファクタリングやレビュー対応をバックグラウンドで並行実行しているとき、`claude agents` の一覧を定期的に見て「Needs input」に上がってきた行だけに対応する運用が基本です。今回の修正により、ツール呼び出しだけで応答テキストを伴わないターン（例えば確認ダイアログの表示のみで完結するツール実行）の直後でも、実際に入力待ちであるセッションが「Working」に紛れて見落とされることがなくなりました。

```bash
claude agents
# Needs input セクションに並んだ行だけを確認して返信すればよい
```

### 通知フックと組み合わせて見落としを防ぐ

`Notification` フックの `agent_needs_input` タイプを利用すると、セッションが入力を必要とした瞬間に外部通知を送ることができます。この修正前は状態が「Working」に戻ってしまうケースがあり、通知後に一覧を確認しても「作業中」に見えて混乱する場合がありましたが、修正後は状態表示と実際のセッションの待機状態が一致するようになりました。

```json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "agent_needs_input",
        "hooks": [{ "type": "command", "command": "notify-send 'Claude Codeが入力待ちです'" }]
      }
    ]
  }
}
```

## 注意点

- この修正はエージェント一覧の表示ロジックに関するバグフィックスであり、新機能の追加ではありません。表示上の状態と実際のセッションの待機状態がずれていた既存のバグが解消されたものです。
- 行のサマリーテキストはHaikuクラスのモデルによって生成され、セッションが稼働中の間は最大15秒に1回、加えて各ターン終了時に1回更新されます。テキストの有無に関わらず、状態そのもの（Needs input / Working など）の判定が今回正しく修正された点に注意してください。
- 同種の不具合として、完了済みの行が「Done」と「Needs your input」の間を行き来する別のフリップ現象も別途修正されています。エージェント一覧の状態表示は継続的に改善が加えられている領域です。
