## 原文（日本語に翻訳）

強制終了されたバックグラウンドセッションが永続的な `git worktree lock` を残してしまう問題を修正しました。定期的なスイープ処理が、所有プロセスが存在しなくなったロックを解放するようになりました。

## 原文（英語）

Fixed killed background sessions leaving a permanent `git worktree lock` behind; the periodic sweep now releases locks whose owning process is gone

## 概要

Claude Codeはバックグラウンドセッションやサブエージェント用に `git worktree` を作成し、実行中は誤って削除されないよう `git worktree lock` を付与します。以前のバージョンでは、そのセッションが `kill -9` やクラッシュなどで異常終了した場合、ロックが解放されないまま残り続け、定期的なクリーンアップ（sweep）処理が対象のworktreeを永遠にスキップしてしまう不具合がありました。v2.1.210では、スイープ処理がロックの所有プロセスがすでに存在しないことを検知し、そのロックを自動的に解放するようになりました。

## 基本的な使い方

### 修正前の問題

```bash
# バックグラウンドセッションを起動
claude --worktree feature-x &
BG_PID=$!

# セッションを強制終了（クラッシュを模擬）
kill -9 $BG_PID

# worktreeの状態を確認
git worktree list --porcelain
# worktree /path/to/repo/.worktrees/feature-x
# HEAD abcdef123456
# branch refs/heads/feature-x
# locked   ← ロックが残ったまま

# 定期スイープが実行されても...
# → ロックされているためスキップされ続ける
# → .worktrees/ ディレクトリが永遠に溜まっていく
```

### 修正後の動作

```bash
# 同様にセッションを強制終了した後、
# Claude Codeの定期スイープが以下を実施:
# 1. ロックされたworktreeを検出
# 2. ロックの所有プロセス（PID）を確認
# 3. プロセスが存在しない場合、ロックを解放
# 4. 通常のクリーンアップ条件（未コミット変更なし等）を満たせば削除

git worktree list --porcelain
# → ロックが解除され、不要なworktreeがクリーンアップされる
```

## 実践例

### 大量のバックグラウンドエージェントを走らせるワークフロー

CI環境やエージェントファームで多数のバックグラウンドセッションを並列実行していると、異常終了によるロック残留が積み重なりやすい状況でした。

```bash
# 複数のエージェントを並列起動
for feature in feature-a feature-b feature-c; do
  claude --worktree "$feature" -p "対応するタスクを実行して" &
done

# 一部がタイムアウトやOOMで強制終了されても、
# 次回の定期スイープで自動的にロックが解放され、
# ディスク容量やworktreeスロットが枯渇しにくくなった
```

### 手動での確認・強制解放（回避策）

自動解放を待てない場合や、修正前バージョンを使っている場合の手動対応。

```bash
# ロックされたworktreeを確認
git worktree list

# 所有プロセスが存在しないことを確認してから手動解放
git worktree unlock /path/to/repo/.worktrees/feature-x

# 未コミット変更がある場合は --force で削除
git worktree remove --force /path/to/repo/.worktrees/feature-x
```

### `--worktree` フラグで作成したセッションとの違い

```bash
# 明示的に --worktree フラグを付けて作成したセッションは、
# このスイープの対象外（自動削除されない）
claude --worktree my-manual-worktree

# 一方、バックグラウンドタスクやサブエージェントが
# 内部的に作成したworktreeはスイープ対象であり、
# 今回の修正でロック残留問題が解消された
```

## 注意点

- この修正はClaude Code v2.1.210で実装されました
- スイープ処理はデフォルトで `cleanupPeriodDays` 設定に基づき、一定期間経過したworktreeのうち、未コミット変更・未追跡ファイル・未プッシュのコミットがないものを対象とします
- ユーザーが明示的に `--worktree` で作成したworktreeは自動削除の対象外です
- 修正前は、プロセスが強制終了された場合にロックだけが残り、通常のクリーンアップが機能しなくなっていました
- 手動でロックを解除する場合は `git worktree unlock`、削除する場合は `git worktree remove --force` を使用できます
- ロックの所有プロセスが実際に生存しているかどうかはPIDの存在確認によって判定されるため、同一PIDが別プロセスに再利用されるような極めて稀なケースでは誤判定の可能性が理論上残ります
