## 原文（日本語訳）

対象のテキストが一意に一致する場合、読み込み後に変更されたファイルに対してEditツールが失敗していた問題を修正。

## 原文（英語）

Fixed the Edit tool failing on files modified after reading when the target text still matches uniquely

## 概要

Claude CodeのEditツールは、安全のため「Claudeが会話内で該当ファイルを読み込んでいること」「`old_string` がファイル内容と完全一致すること」「その一致が一意であること」という3つのチェックを通過しないと編集が適用されません。これまでは、Claudeがファイルを読み込んだ後に外部プロセス（フォーマッタや他のツール、あるいはユーザー自身の編集など）がそのファイルをディスク上で変更すると、たとえ置換対象の文字列が現在のファイル内容と一意に一致していても編集が拒否されていました。今回のアップデートで、ディスク上の変更後でも `old_string` が現在の内容と一意に一致していれば編集できるようになりました。

## 基本的な使い方

通常の `Edit` ツール呼び出しの動作自体は変わりません。Claudeがファイルを読み込んだ後、他プロセスがそのファイルを書き換えても、置換対象の文字列がファイル内に一意に存在していれば、そのままEditが成功します。

```text
src/utils.ts の calculateTotal 関数のバグを直して
```

Claudeが `src/utils.ts` を読み込んだ後にlinterの自動整形（`prettier --write` など）が走ってファイルが変わっていても、修正対象のコード片が一意に残っていればEditはそのまま適用されます。

## 実践例

### フォーマッタやlintの自動修正と並行して作業する場合

保存時に自動整形が走るエディタや `husky` の pre-commit フックなどを使っている環境では、Claudeがファイルを読み込んだ直後にインデントや改行コードが自動的に書き換わることがあります。

```text
このファイルのimport文を整理して
```

以前は「ファイルが変更されています」という理由でEditが失敗し、Claudeに再読み込みを促す必要がありましたが、修正後は対象文字列が一意に一致する限り、余計な再読み込みなしでそのまま編集が完了します。

### 複数のエージェントやプロセスが同じリポジトリを触る場合

CI連携やバックグラウンドタスク、別のターミナルでのGit操作などにより、Claudeの読み込み後にファイルが更新されるケースは珍しくありません。

```text
config.yaml のタイムアウト値を30に変更して
```

編集対象の行が他の変更と衝突せず一意に特定できる場合は、これまでのように「ファイルが変更されたため再読み込みしてください」というエラーで止まることなく編集が進みます。ただし、置換対象が曖昧になったり一致しなくなったりした場合は、引き続きClaudeが自動的にファイルを再読み込みしてから編集します。

## 注意点

- `old_string` が現在のファイル内容と完全一致し、かつ一意に特定できる場合のみ適用される緩和です。一致しない場合や複数箇所に一致する場合は、従来どおりClaudeが再読み込みしてから編集します。
- 置換後、そのファイルには読み込み時点から他の変更が含まれていることをClaudeは認識し、周辺内容に依存する後続の編集の前には再読み込みを行います。
- この挙動の緩和はClaude Code v2.1.208以降が対象です。それより前のバージョンでは、会話内で読み込んでいない、またはディスク上で変更されたファイルへの編集は常に拒否されていました。
