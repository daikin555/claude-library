## 原文（日本語訳）

Claudeが作業中に送信されたメッセージが、`--max-turns` の上限に達してターンが終了した際にサイレントに失われてしまう問題を修正。

## 原文（英語）

Fixed a message sent while Claude was working being silently lost when the turn ended at the `--max-turns` limit

## 概要

`--max-turns` は、Claude Codeが1回のセッション内で実行できるツール呼び出し（ターン）の回数に上限を設けるフラグで、CI/CDや自動化パイプラインなどの非対話的な実行環境で暴走を防ぐために利用されます。今回の修正前は、Claudeが作業を進めている最中にユーザー（またはキューイングされた処理）からメッセージが送られてきた場合、ちょうど `--max-turns` の上限に達してターンが終了するタイミングと重なると、そのメッセージが警告やエラーもなく破棄されてしまう不具合がありました。今回のアップデートでこの取りこぼしが修正され、上限到達時でも送信されたメッセージが失われずに扱われるようになりました。

## 基本的な使い方

`--max-turns` は `claude -p`（非対話モード）と組み合わせて、1回のやり取りで許容するツール呼び出し回数の上限を指定します。

```bash
claude -p "リンティングエラーを修正して" --max-turns 5
```

上記の例では、Claudeが5ターン（5回のツール呼び出し）以内に作業を終わらせようとし、上限に達すると処理を打ち切ります。

## 実践例

### CIパイプラインでの暴走防止

CI上で自動修正タスクを走らせる際、無限ループやトークンの過剰消費を防ぐために `--max-turns` を設定しておくのが定石です。

```bash
claude -p "テストが失敗している原因を調査し、可能なら修正して" \
  --max-turns 10 \
  --output-format json
```

以前はこのような自動実行中に、外部プロセスから追加の指示メッセージを送り込んだ場合、上限到達のタイミングによってはそのメッセージがログにも残らず静かに消えてしまうことがありました。修正後は、上限到達と重なった場合でもメッセージが正しく扱われるため、自動化フロー側でメッセージが握りつぶされていないか疑心暗鬼になる必要がなくなります。

### 長時間タスクの途中で追加指示を送るワークフロー

SDKやスクリプトからClaude Codeのセッションに対して、作業中に追加のフォローアップメッセージを送信するようなワークフローでも、この修正の恩恵があります。

```bash
claude -p "大規模なリファクタリングを進めて" --max-turns 30
# 作業中に別プロセスから追加のメッセージをキューイングして送信
# → 以前はターン上限到達と重なるとメッセージが消失することがあった
```

このようなケースでは、メッセージが確実に処理される（あるいは少なくとも失われず何らかの形で扱われる）ことが保証されるようになったため、途中経過の指示出しをより信頼して行えます。

## 注意点

- `--max-turns` の上限に達すると、そこで作業自体は打ち切られます（今回の修正はあくまで「送信済みメッセージの消失」を防ぐものであり、上限そのものを緩和・変更するものではありません）。
- `--max-turns` は現時点で `claude --help` の出力や一部の公式ドキュメントページに明記されていないことがあり、CLIオプションとしては存在するものの記載が薄い状態が指摘されています。挙動の詳細は変更される可能性があるため、重要な自動化パイプラインでは実際の挙動を都度確認することが推奨されます。
- 上限到達時にエラー終了する設計のスクリプト・ワークフローでは、引き続き try/catch などでハンドリングしておく必要があります。
