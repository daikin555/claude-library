## 原文（日本語訳）

ユーザースコープで保存する際、ワークフロー保存ダイアログが `CLAUDE_CONFIG_DIR` で指定された場所ではなく、常に `~/.claude/workflows/` を表示していた不具合を修正した。

## 原文（英語）

Fixed the workflow save dialog showing `~/.claude/workflows/` instead of the `CLAUDE_CONFIG_DIR` location for user-scope saves

## 概要

Claude Codeの[Dynamic workflows](https://code.claude.com/docs/en/workflows)機能では、`/workflows` から実行結果を選択し `s` キーを押すことで、そのワークフローをコマンドとして保存できます。保存先はプロジェクト共有の `.claude/workflows/` か、個人用の `~/.claude/workflows/` から選択でき、`CLAUDE_CONFIG_DIR` 環境変数を設定している場合、個人用ワークフローは実際にはその配下の `workflows/` ディレクトリに保存されます。しかしv2.1.208より前は、保存ダイアログの表示パスが `CLAUDE_CONFIG_DIR` の設定を反映せず、常に `~/.claude/workflows/` と表示されてしまい、実際の保存先と画面表示が食い違っていました。今回の修正で、ダイアログには解決済みの実際のパスが正しく表示されるようになりました。

## 基本的な使い方

`/workflows` でワークフロー実行結果を選択し、`s` キーで保存ダイアログを開きます。`Tab` キーでプロジェクトスコープと個人スコープを切り替え、`Enter` で保存します。

```text
/workflows
# 実行結果を選択して s キーを押す
# Tab でプロジェクト用 (.claude/workflows/) と
# 個人用 (~/.claude/workflows/ または $CLAUDE_CONFIG_DIR/workflows/) を切り替え
# Enter で保存
```

## 実践例

### CLAUDE_CONFIG_DIRを設定したチーム開発環境での保存

セキュリティポリシーなどの理由で `CLAUDE_CONFIG_DIR` を独自の場所に設定しているチームでは、個人用ワークフローの保存先も自動的にその配下になります。

```bash
export CLAUDE_CONFIG_DIR="/opt/claude-config"
```

この状態で `/workflows` から保存すると、v2.1.208以降ではダイアログに `/opt/claude-config/workflows/` のように正しい保存先パスが表示され、実際の保存場所と一致します。

### 複数プロファイルを使い分ける開発者の運用

案件ごとに `CLAUDE_CONFIG_DIR` を切り替えて複数の設定プロファイルを使い分けている場合、どのプロファイルにワークフローが保存されるのかをダイアログの表示で正確に確認できるようになります。

```bash
# プロファイルAで作業
CLAUDE_CONFIG_DIR="$HOME/.claude-profile-a" claude
# /workflows → s で保存すると、ダイアログに
# ~/.claude-profile-a/workflows/ が表示される
```

## 注意点

- このバグは表示上の問題であり、v2.1.208より前でも `CLAUDE_CONFIG_DIR` を設定していれば実際の保存先自体は正しい場所（設定したディレクトリ配下）になっていました。混乱を招く表示のみが問題でした。
- プロジェクトスコープ（`.claude/workflows/`）の保存には影響しません。影響があるのはユーザー（個人）スコープの保存表示のみです。
- Dynamic workflows自体はv2.1.154以降が必要な機能です。本修正はv2.1.208以降で適用されます。
