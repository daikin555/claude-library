## 原文（日本語訳）

Windows環境において、読み取り専用コマンドがネットワークパスにアクセスする際、権限確認プロンプトなしに実行されてしまう問題を修正しました。

## 原文（英語）

Fixed read-only commands on Windows accessing network paths without a permission prompt

## 概要

Claude Codeは`ls`や`cat`に相当する読み取り専用コマンドについては、通常のパーミッションモードでも確認プロンプトなしに実行を許可しています。しかしWindows環境では、UNCパス（`\\server\share\...`）やマップされたネットワークドライブなど、ネットワーク上のパスに対してこれらの読み取り専用コマンドが実行される場合にも確認プロンプトがスキップされてしまい、意図せずネットワーク上のリソースへアクセスしてしまう可能性がありました。今回の修正により、Windowsでのネットワークパスへのアクセスは読み取り専用コマンドであっても適切に権限確認の対象となるようになりました。

## 基本的な使い方

この修正はセキュリティ関連の挙動改善であり、追加の設定は不要です。Windows環境で通常どおりClaude Codeを利用するだけで、修正後の挙動が適用されます。

```
> ネットワークドライブ Z: のファイル一覧を見せて
```

上記のような依頼でネットワークパスに対する読み取り専用コマンドが実行される場合、修正後は必要に応じて権限確認プロンプトが表示されます。

## 実践例

### UNCパス上のファイルを確認する

```
> \\fileserver\shared\reports のファイル一覧を確認して
```

このような操作は、修正前はネットワークパスであっても読み取り専用コマンドとして無条件に実行されていましたが、修正後はローカルファイルシステム外のリソースへのアクセスとして扱われ、必要に応じて確認が求められます。

### マップされたネットワークドライブを参照する

```
> Zドライブの共有フォルダにあるログファイルをcatして
```

社内共有ドライブなどマップされたネットワークドライブへのアクセスについても、同様に権限確認の対象になります。

### 許可ルールを事前に設定する

頻繁にアクセスする信頼できるネットワークパスがある場合は、`.claude/settings.json`で明示的に許可ルールを設定しておくことで、毎回のプロンプトを避けられます。

```json
{
  "permissions": {
    "allow": [
      "Read(//fileserver/shared/**)"
    ]
  }
}
```

## 注意点

- この修正はWindows環境に特有の問題への対応です。macOSやLinuxでの読み取り専用コマンドの扱いには変更はありません。
- ネットワークパスへのアクセスは、ローカルファイルシステムへのアクセスと異なり、外部の共有リソースに影響が及ぶ可能性があるため、権限確認の対象とすることはセキュリティ上望ましい挙動です。
- 業務で頻繁に使うネットワーク共有がある場合は、`allow`ルールを設定ファイルに追加しておくと、都度の確認プロンプトを避けつつ安全に運用できます。
- パスのパターン指定では、設定ファイルからの相対パスと絶対パスの扱いに注意してください（先頭に`//`を付けると絶対パス指定になります）。
