## 原文（日本語に翻訳）

後から出現した `.claude/*` シンボリックリンクが、サンドボックスの書き込み拒否リストに反映（reconcile）されていなかった問題を修正しました。

## 原文（英語）

Fixed late-appearing `.claude/*` symlinks not being reconciled into the sandbox deny-write list

## 概要

Claude Codeのサンドボックス機能は、`.claude/settings.json` などClaude Code自身の設定ファイルへの書き込みを既定で拒否し、サンドボックス化されたコマンドが自身のポリシーを書き換えられないようにしています。しかし以前のバージョンでは、サンドボックス初期化後に新しく `.claude/` 配下へシンボリックリンクが作成された場合、そのシンボリックリンクが書き込み拒否リスト（deny-write list）へ反映されず、意図せず書き込み可能な抜け道になってしまう可能性がありました。v2.1.210では、後から出現したシンボリックリンクも定期的に検出・反映されるように修正されています。

## 基本的な使い方

### 修正前の問題

```bash
# サンドボックスを有効にしてClaude Codeを起動
claude --sandbox

# セッション中に、何らかのプロセス（プラグインやスクリプト）が
# .claude/ 配下に新しいシンボリックリンクを作成
ln -s /tmp/writable-settings.json .claude/settings.local.json

# 修正前:
# .claude/settings.local.json はシンボリックリンクだが、
# サンドボックス初期化後に出現したため
# deny-writeリストに反映されておらず、
# リンク先（/tmp/writable-settings.json）への書き込みが
# 意図せず許可されてしまう可能性があった
```

### 修正後の動作

```bash
# サンドボックスは後から出現したシンボリックリンクも検出し、
# .claude/ 配下を指すものであれば
# deny-writeリストへ反映（reconcile）する

ln -s /tmp/writable-settings.json .claude/settings.local.json

# 修正後:
# シンボリックリンク自体、およびそのリンク先の双方が
# 書き込み拒否の対象として扱われる
```

## 実践例

### プラグインやツールが動的にシンボリックリンクを作る場合

```bash
# 一部のプラグインやセットアップスクリプトが、
# 共有設定を .claude/ 配下にシンボリックリンクとして
# 展開するケース

/plugin install some-tooling-plugin
# → プラグインが .claude/settings.json.link のような
#   シンボリックリンクを追加で作成することがある

# 修正後は、このようなリンクが後から追加されても
# 適切に書き込み拒否リストへ組み込まれるため、
# サンドボックス化されたコマンドから設定を
# 書き換えられてしまうリスクが防がれる
```

### 複数worktreeで.claude設定を共有する構成

```bash
# 複数のworktreeで同じ.claude設定を共有するために
# シンボリックリンクを使う運用
cd .worktrees/feature-x
ln -s ../../.claude/settings.json .claude/settings.json

# サンドボックス化されたセッションでこのworktreeに入っても、
# シンボリックリンク経由での設定書き換えは拒否される
claude --sandbox
```

### 手動でのサンドボックス設定確認

```bash
# 現在のdenyWrite設定を確認する例
cat .claude/settings.json
# {
#   "sandbox": {
#     "filesystem": {
#       "denyWrite": ["**/.claude/settings*.json"]
#     }
#   }
# }

# 後から作成したシンボリックリンクが
# このパターンに合致する場合、
# リンク自体・リンク先ともに書き込みが拒否される
```

## 注意点

- この修正はClaude Code v2.1.210で実装されました
- サンドボックスのdeny系ルールは、シンボリックリンク自体かそのリンク先のいずれかが対象パターンに一致すれば拒否されます（allow系ルールとは異なり、両方が一致する必要はありません）
- サンドボックスは既定で全スコープの `.claude/settings*.json` およびマネージドセッティングディレクトリへの書き込みを拒否し、サンドボックス化されたコマンドが自身のポリシーを書き換えられないようにしています
- この修正はあくまで「後から出現したシンボリックリンクの反映漏れ」を解消するものであり、`sandbox.filesystem.denyWrite` / `denyRead` / `allowRead` 自体の設定方法は変わっていません
- サンドボックス設定を独自にカスタマイズしている場合は、シンボリックリンク経由の抜け道が生じていないか、`git status` や `ls -la .claude/` で定期的に確認することを推奨します
