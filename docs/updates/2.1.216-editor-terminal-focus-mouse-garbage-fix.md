## 原文（日本語訳）

`/memory`、`/plan`、`/keybindings`、またはCtrl+Gで外部GUIエディタを開いている間に、ターミナル上でマウスやフォーカスに関するゴミ（表示崩れ）が発生する問題を修正しました。また、`/memory`はエディタが閉じるのを待たなくなりました。

## 原文（英語）

Fixed mouse and focus garbage in the terminal while a GUI editor from `/memory`, `/plan`, `/keybindings`, or Ctrl+G is open; `/memory` no longer waits for the editor to close

## 概要

Claude Codeでは`/memory`、`/plan`、`/keybindings`コマンドやCtrl+Gのショートカットから、VS Codeなどの外部GUIエディタを起動してファイルを編集できます。従来はこの外部エディタが開いている間、ターミナルのマウストラッキングモードやフォーカスイベントの状態がうまく引き継がれず、エディタを閉じてターミナルに戻った際にマウス操作の残骸やフォーカスの不整合による表示崩れが発生することがありました。今回の修正でこの状態管理が改善され、あわせて`/memory`コマンドはエディタのプロセス終了を待たずにターミナル操作へ復帰するようになりました。

## 基本的な使い方

この修正は挙動の自動改善であり、ユーザー側での設定変更は不要です。`/memory`、`/plan`、`/keybindings`を実行するか、Ctrl+Gを押して外部エディタを開くだけで、修正後の挙動が適用されます。

```
> /memory
```

上記を実行すると設定済みのGUIエディタ（例: VS Code）が開きますが、v2.1.216以降は`/memory`自身がエディタの終了を待たずにターミナルの操作をすぐに再開できるようになりました。

## 実践例

### /memoryでCLAUDE.mdを編集する

```
> /memory
```

外部エディタが開いている間もターミナルの表示は崩れず、エディタ側での編集が終わるのを待たずに他のターミナル操作（スクロールやマウス操作）を継続できます。

### Ctrl+Gで一時ファイルを編集する

プロンプト入力中にCtrl+Gを押すと、現在の入力内容が一時ファイルとして外部エディタで開きます。編集を終えてエディタを閉じると、修正前はマウス位置やフォーカス状態がずれて表示が乱れることがありましたが、修正後はクリーンにターミナルへ復帰します。

```
（プロンプト入力中に）Ctrl+G
```

### /keybindingsでキー割り当てを確認・編集する

```
> /keybindings
```

キーバインド設定ファイルが外部エディタで開きますが、同様にマウス/フォーカスまわりの表示崩れが解消されています。

## 注意点

- この修正はターミナルの表示挙動に関するものであり、機能面の変更はありません。
- 外部GUIエディタの設定は`$EDITOR`環境変数や設定ファイルで行います。エディタが正しく設定されていない場合はこの修正の恩恵を受けられない点に注意してください。
- `/memory`がエディタの終了を待たなくなったため、エディタでの保存前にターミナルへ戻って別の操作をしても、メモリファイルへの反映は保存タイミングに依存します。編集内容を確実に反映させるには、エディタ側で保存操作を完了させてください。
- ターミナルエミュレータの種類によってはマウストラッキングやフォーカスイベントの実装が異なるため、環境によって体感できる改善幅が異なる場合があります。
