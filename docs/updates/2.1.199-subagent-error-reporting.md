## 原文（日本語に翻訳）

サブエージェントが利用上限到達などのAPIエラーを成功した結果として報告してしまう問題を修正 — 現在はエラーが親エージェントに正しく報告されるようになりました

## 原文（英語）

Fixed subagents reporting API errors (e.g. usage limit reached) as successful results — the error is now reported to the parent agent

## 概要

サブエージェントが利用上限（usage limit）到達などのAPIエラーに遭遇した際、これまではそれをあたかも成功した結果であるかのように親エージェントへ報告してしまうことがありました。これにより、親エージェントはサブエージェントのタスクが実際には失敗していたにもかかわらず成功したと誤認し、誤った前提で後続の処理を進めてしまう可能性がありました。この修正により、APIエラーは正しくエラーとして検出され、親エージェントへ明示的に報告されるようになりました。

## 基本的な使い方

Before（修正前）:
```
親エージェント: サブエージェントにタスクを委譲
サブエージェント: （利用上限に到達しAPIエラーが発生）
→ それにもかかわらず「タスクが完了しました」という
  成功結果として親エージェントに報告されてしまう
```

After（修正後）:
```
親エージェント: サブエージェントにタスクを委譲
サブエージェント: （利用上限に到達しAPIエラーが発生）
→ エラーであることが正しく検出され、
  「APIエラーによりタスクが完了できませんでした」と
  親エージェントに報告される
```

## 実践例

### 利用上限（usage limit）に到達した状態での並列タスク実行

複数のサブエージェントを並列実行している最中に利用上限に達した場合、修正前は一部のサブエージェントが「成功」と誤報告し、実際には未完了のタスクが完了扱いになってしまうことがありました。修正後は正しくエラーとして扱われるため、親エージェントは失敗したタスクを正確に把握し、必要に応じて再実行や別対応を判断できます。

### 自動化されたマルチエージェントワークフロー

CI的にサブエージェントを連鎖させて自動処理を行うワークフローでは、誤った成功報告があると後続のステップが不正な前提で進んでしまいます。修正後はエラーが正しく伝播するため、ワークフロー全体の信頼性が向上します。

### レビューやコード生成タスクの委譲

コードレビューや生成をサブエージェントに任せている場合、APIエラーが成功として扱われると「レビュー済み」「生成済み」と誤認したまま作業を進めてしまうリスクがありましたが、修正後はエラーが明示されるため、誤った成果物を見逃すリスクが減ります。

## 注意点

- 本修正はエラーの「報告」が正しく行われるようにするものであり、利用上限そのものを回避するものではありません。上限に達した場合は時間をおくかプランの見直しが必要です。
- 親エージェント側でエラー報告を受け取った際に、どのようにリカバリー（再試行・スキップなど）するかは引き続きワークフロー設計側での考慮が必要です。
- 2.1.199の別項目「サブエージェントがレート制限/サーバーエラーで中断された際に部分的な作業を返す」修正と合わせて、サブエージェントのエラー伝播全般の信頼性向上の一環です。
