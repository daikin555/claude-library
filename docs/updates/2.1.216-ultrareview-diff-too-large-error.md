## 原文（日本語訳）

`/ultrareview` のdiff過大エラーを改善し、設定されている上限値、実測されたdiffサイズ、そして差分に最も寄与している大きなファイルを表示するようにしました。

## 原文（英語）

Improved the `/ultrareview` diff-too-large error to show configured limits, measured diff size, and largest contributing files

## 概要

`/ultrareview`（クラウド上でマルチエージェントが深いコードレビューを行うコマンド）は、あまりに大きな差分を送信しようとするとエラーになりますが、これまでは「diffが大きすぎる」という情報しか得られず、何を減らせば通るのか分かりにくいことがありました。今回の改善により、エラーメッセージに「設定されている上限値」「実際に測定されたdiffのサイズ」「差分の中で特に大きい寄与をしているファイル一覧」が表示されるようになり、原因の特定と対処がその場で行いやすくなりました。

## 基本的な使い方

```bash
/ultrareview
```

差分が大きすぎる場合、以下のような情報を含むエラーが表示されます（イメージ）。

```
Error: Diff too large for /ultrareview
  Configured limit: 5,000 lines
  Measured diff size: 12,430 lines
  Largest contributing files:
    - dist/bundle.js       (+8,210 / -0)
    - package-lock.json    (+2,145 / -320)
    - src/generated/api.ts (+890 / -12)
```

このメッセージから、どのファイルを除外・分割すればレビューを実行できるかが一目で分かります。

## 実践例

### 生成ファイル・ロックファイルを除外してから再実行

エラーで示された「最大寄与ファイル」がビルド成果物やロックファイルであれば、それらをコミットから分離したり `.gitignore` に含めたりしてから再度実行します。

```bash
# 例: package-lock.json など生成物が差分の大半を占めていた場合
git diff --stat | sort -k3 -n -r | head

# 該当ファイルを除いた形でレビュー対象を絞り込んでから再実行
/ultrareview
```

### 変更を複数のPR・ブランチに分割する

実測サイズが設定上限を大きく超えている場合は、変更を意味のある単位で分割することを検討します。

```bash
# 大きな機能追加を複数の論理的な単位に分割
git checkout -b feature/part-1
# ... 一部の変更のみをコミット ...
/ultrareview
```

### 上限値の設定を確認する

表示された「Configured limit」を見て、必要であれば管理者に上限緩和の相談をするか、レビュー対象を絞り込みます。

```bash
# エラーメッセージの Configured limit を確認し、
# チームのポリシーとして妥当かどうかを判断する
```

## 注意点

- この改善はエラーメッセージの情報量を増やすものであり、上限値そのものを変更するものではありません
- 「最大寄与ファイル」にはロックファイルやビルド成果物、自動生成コードが含まれやすいため、まずそれらの除外を検討するとよいでしょう
- `/ultrareview` はエクストラ使用量として課金される場合があるため、diffを絞り込んでから実行する方がコスト面でも有利です
- 大規模な変更は複数の論理的なコミット・PRに分割してレビューする方が、レビュー精度の面でも望ましい場合があります
