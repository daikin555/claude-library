## 原文（日本語訳）

多数のMCPツールを持つprint/SDKセッションにおいて、ツールプールの組み立てをキャッシュすることで、ツール呼び出しごとのCPUオーバーヘッドを削減した（ツール数が多い場合、ツールラウンドが最大7倍高速化）。

## 原文（英語）

Reduced per-tool-call CPU overhead in print/SDK sessions with many MCP tools by caching tool-pool assembly (up to 7x faster tool rounds at high tool counts)

## 概要

Claude Codeを `claude -p`（printモード）やClaude Agent SDK経由で利用する場合、多数のMCPサーバー・多数のツールを接続していると、これまではツール呼び出しのラウンドごとに利用可能なツール一覧（ツールプール）を毎回組み立て直しており、これがCPUオーバーヘッドの原因になっていました。今回の変更でツールプールの組み立て結果がキャッシュされるようになり、ツール数が多い環境ではツールラウンドの処理が最大7倍高速化しました。

## 基本的な使い方

これはユーザーが明示的に呼び出す機能ではなく、内部的な処理の最適化です。特別な操作は不要で、多数のMCPツールを接続したprint/SDKセッションで自動的に恩恵を受けられます。

```bash
claude -p "MCP経由で接続した複数ツールを使ってタスクを実行して"
```

## 実践例

### 多数のMCPサーバーを接続したSDKエージェントの高速化

データベース、Issueトラッカー、監視ダッシュボードなど複数のMCPサーバーを同時に接続し、多数のツールが利用可能な状態でClaude Agent SDKからエージェントを繰り返し呼び出すようなワークロードでは、ツール呼び出しラウンドごとの組み立てコストが積み重なっていました。ツールプールのキャッシュにより、ツール数が多いほど高速化の効果が大きくなります。

```python
# Claude Agent SDK での例（イメージ）

```

### CI/バッチ処理でのprintモード実行の高速化

`claude -p` をCIパイプラインやバッチ処理に組み込み、多くのMCPツールを有効化した状態で繰り返し実行しているケースでは、1回あたりのツールラウンド処理時間が短縮されることで、パイプライン全体の実行時間短縮につながります。

## 注意点

- 効果が顕著に現れるのは、接続しているMCPツールの数が多い（多数のMCPサーバーを同時利用している）場合です。ツール数が少ない環境では体感差は小さい可能性があります。
- 通常の対話型セッション（インタラクティブモード）でも内部的な恩恵はありますが、特に `-p`（print）モードやSDK経由での連続実行で効果が大きいとされています。
- ユーザー側での設定変更は不要です。
