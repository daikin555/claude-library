## 原文（日本語に翻訳）

`ANTHROPIC_BASE_URL` と `ANTHROPIC_AUTH_TOKEN` を使用したカスタムAPIゲートウェイ経由でデーモンが起動した場合に、`claude agents` ワーカーが `401 Invalid bearer token` で失敗する問題を修正しました。

## 原文（英語）

Fixed `claude agents` workers failing with `401 Invalid bearer token` when the daemon was started from a shell with a custom API gateway via `ANTHROPIC_BASE_URL` and `ANTHROPIC_AUTH_TOKEN`

## 概要

企業内プロキシや代替APIゲートウェイ（`ANTHROPIC_BASE_URL` 環境変数で設定）を使用している環境で、`claude agents` のサブエージェントワーカーが認証エラー（401）で失敗する問題が修正されました。デーモン起動時のカスタム認証設定がワーカープロセスに正しく引き継がれるようになりました。

## 基本的な使い方

カスタムAPIゲートウェイを使用する場合の設定:

```bash
# カスタムゲートウェイの設定
export ANTHROPIC_BASE_URL="https://your-api-gateway.company.com"
export ANTHROPIC_AUTH_TOKEN="your-custom-token"

# Claude Code を起動（この設定がワーカーにも引き継がれるようになった）
claude
```

## 実践例

### 企業内APIプロキシを使用する環境

```bash
# .bashrc や .zshrc での設定例
export ANTHROPIC_BASE_URL="https://api-proxy.internal.company.com/v1"
export ANTHROPIC_AUTH_TOKEN="internal-bearer-token-12345"

# 設定後にClaude Codeを起動
claude

# 以前の問題: claude agents でサブエージェントを起動すると
# ワーカーが独自に認証を試み、401エラーが発生していた
```

### AWS/GCP/Azureのプロキシ経由での使用

```bash
# AWS Bedrockプロキシ経由の設定
export ANTHROPIC_BASE_URL="https://bedrock-proxy.us-east-1.amazonaws.com"
export ANTHROPIC_AUTH_TOKEN=$(aws sts get-session-token --output text)

claude

# 修正後: サブエージェントも同じゲートウェイとトークンを使用
```

### マルチエージェントワークフローでの動作確認

```bash
# カスタムゲートウェイ設定後
export ANTHROPIC_BASE_URL="https://your-gateway.com"
export ANTHROPIC_AUTH_TOKEN="your-token"

# サブエージェントを使うワークフローを実行
claude
> run a workflow to analyze all TypeScript files
# → 以前は 401 で失敗。修正後は正常に実行される
```

## 注意点

- `ANTHROPIC_BASE_URL` はAPIエンドポイントのベースURLを指定します（末尾のスラッシュなし）
- `ANTHROPIC_AUTH_TOKEN` はカスタムゲートウェイ用のBearer tokenです（Anthropic公式のAPIキーとは異なる場合があります）
- この問題はデーモンモードで起動したときに特に発生しやすかった（バックグラウンドデーモンが環境変数を引き継がなかった）
- 公式Anthropic APIを直接使用する場合（`ANTHROPIC_API_KEY` のみ設定）はこの問題の影響を受けません
