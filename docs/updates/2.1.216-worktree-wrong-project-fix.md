## 原文（日本語訳）

作業ディレクトリが選択したプロジェクトと一致しない場合に、worktreeセッションが別プロジェクトの残存worktreeに着地してしまう問題を修正しました。

## 原文（英語）

Fixed worktree sessions landing in another project's leftover worktree when the working directory did not match the selected project

## 概要

複数のプロジェクトでworktreeを使い分けている環境において、現在の作業ディレクトリと実際に選択したプロジェクトが一致していないケースで、Claude Codeが誤って別プロジェクトの「使い残し（削除されずに残っていた）」worktreeにセッションを起動してしまう不具合がありました。この修正により、プロジェクトの選択とworktreeの紐付けが正しく検証されるようになり、意図しないプロジェクトのファイルに対して作業してしまう事故が防止されます。

## 基本的な使い方

この修正もバグ修正のため、特別な設定は不要です。プロジェクトを切り替えながらworktreeセッションを開始するだけで、修正後の正しい挙動になります。

```bash
# プロジェクトA配下ではないディレクトリから
# プロジェクトAのworktreeセッションを開始しようとした場合でも
cd /somewhere/else
claude --worktree feature-y
```

修正後は、作業ディレクトリと選択されたプロジェクトの対応関係が正しく検証され、意図した（あるいは新規の）worktreeに着地するようになります。

## 実践例

### 複数リポジトリを行き来する開発フロー

複数のリポジトリでそれぞれworktreeを使っている場合、以前削除し忘れたworktreeが残っていると、それが別プロジェクトのセッション開始時に誤って再利用されるリスクがありました。

```bash
# プロジェクトA用のworktreeがすでに存在
ls .claude/worktrees/

# プロジェクトBのディレクトリから作業を開始
cd ~/projects/project-b
claude --worktree hotfix
```

修正前はこのようなケースで、状況によっては `project-a` 側の残存worktreeにアタッチしてしまうことがありましたが、修正後はプロジェクトBに対応する新しい（または正しい）worktreeが使われます。

### 不要なworktreeのクリーンアップ確認

念のため、定期的に不要になったworktreeを確認・整理することも推奨されます。

```bash
# 各プロジェクトのworktree一覧を確認
git worktree list

# 不要なworktreeを削除
git worktree remove .claude/worktrees/old-feature
```

## 注意点

- この不具合は特に、複数プロジェクトを頻繁に切り替えながら `--worktree` を使う開発者に影響していました
- 修正後も、意図しないworktreeの残存を避けるため、不要になったworktreeはこまめに削除することが推奨されます
- 作業ディレクトリとプロジェクトルートの対応関係が曖昧な場合は、明示的にプロジェクトディレクトリへ `cd` してからセッションを開始することでリスクをさらに下げられます
