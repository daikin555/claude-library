## 原文（日本語訳）

ヘッドレスセッションにおいて、SessionStartフックの実行中にフックイベントがストリーミングされない不具合を修正。これにより、フックの実行中にリモートワーカーがアイドル状態とみなされ、途中で強制終了（reap）されてしまうことがあった。

## 原文（英語）

Fixed hook events not streaming during SessionStart hooks in headless sessions, which could cause remote workers to be idle-reaped mid-hook

## 概要

SessionStartフックは、Claude Codeのセッションが開始・再開されたタイミングで一度だけ実行されるフックで、Gitの状態やプロジェクト固有のコンテキストをセッション開始時に自動で注入する用途などに使われます。`-p`（headless）モードや`--output-format stream-json`によるリモート実行環境では、実行中の処理が生きていることを示すためにイベントストリームが継続的に流れている必要がありますが、従来はSessionStartフックの実行中にこのイベントストリーミングが止まってしまう不具合がありました。その結果、フックの処理に時間がかかる場合、リモートのワーカープロセスが「アイドル状態」と誤認識され、フック完了前にプロセスごと強制終了（idle-reap）されてしまうことがありました。今回の修正により、SessionStartフックの実行中もイベントが継続的にストリーミングされるようになり、こうした誤った強制終了が発生しなくなりました。

## 基本的な使い方

SessionStartフックは`.claude/settings.json`（またはユーザー・ローカル設定）の`hooks.SessionStart`に登録します。

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup",
        "hooks": [
          {
            "type": "command",
            "command": "scripts/inject-context.sh"
          }
        ]
      }
    ]
  }
}
```

ヘッドレスモードでこのセッションを実行する場合は、次のようにストリーミング形式の出力を指定します。

```bash
claude -p "リリースノートを要約して" --output-format stream-json --verbose
```

今回の修正により、`inject-context.sh`のようなSessionStartフックの実行に数秒〜数十秒かかる場合でも、その間イベントストリームが途切れなくなり、リモート実行基盤側がセッションを非アクティブと誤判定して強制終了することがなくなります。

## 実践例

### CI/CDパイプラインでのヘッドレス実行

CIジョブの中でリポジトリの状態確認やチケット情報の取得を行うSessionStartフックを設定しているケースでは、外部API呼び出しなどでフックの完了に時間がかかることがあります。従来はこの間にストリームが止まり、CIランナーやオーケストレーション基盤がセッションをタイムアウト・強制終了してしまう可能性がありましたが、修正後はイベントが継続的に送出されるため、フックが安定して完了するまで待機できます。

```bash
claude -p "変更差分をレビューして" \
  --output-format stream-json --verbose
```

### リモートワーカー上での長時間SessionStartフック

自前のリモートオーケストレーション基盤（例：複数のワーカーコンテナでheadlessなClaude Codeセッションを起動し、アイドル時間が一定を超えたワーカーを自動的に回収する仕組み）を運用している場合、SessionStartフックが重い処理（大規模リポジトリのインデックス作成や外部サービスへの問い合わせなど）を行っていると、ストリーミングが途絶えることでワーカー監視側から「応答なし」と判断され、セッションごと回収（reap）されてしまうリスクがありました。今回の修正はこの経路を修正するものなので、重めのSessionStartフックを使っているリモートワーカー環境ほど恩恵が大きくなります。

## 注意点

- この修正はSessionStartフックの実行中にイベントストリームが途切れる問題への対処であり、フック自体の処理時間そのものを短縮するものではありません。フックが恒常的に長時間かかる場合は、別途フックの軽量化を検討してください。
- `stream-json`形式で出力を受け取るには`--verbose`の併用が必要です。`--output-format stream-json`だけを指定しても意図した形式で出力されない点に注意してください。
- リモートワーカーの回収（idle-reap）ロジック自体は各実行基盤（自前のオーケストレーションシステムやCI基盤）側の実装に依存します。今回の修正はClaude Code側でイベントストリーミングが途切れないようにするものであり、外部監視の閾値設定が極端に短い場合は依然として注意が必要です。
