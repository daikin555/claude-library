## 原文（日本語訳）

ファイル編集用の読み込みキャッシュの上限を、最大1,000個のファイル全体をピン留めする方式から16MBに変更し、メモリ使用量を削減した。

## 原文（英語）

Reduced memory usage by bounding the file edit read cache to 16 MB instead of pinning up to 1,000 full files

## 概要

Claude Codeは `Edit` ツールでファイルを編集する際、直前に読み込んだファイル内容をキャッシュして整合性チェックなどに利用しています。これまでこのキャッシュは、ファイルサイズに関係なく最大1,000個のファイル全体をメモリ上にピン留めする方式だったため、大きなファイルを多数扱うセッションではメモリ使用量が想定以上に膨らむことがありました。今回の変更でキャッシュの上限がファイル数ではなく合計16MBというサイズ基準になり、メモリ使用量が予測可能な範囲に収まるようになりました。

## 基本的な使い方

これはユーザーが明示的に呼び出す機能ではなく、内部的なメモリ管理の改善です。特別な操作は不要で、アップデート後は自動的に適用されます。

```text
Claudeが Read → Edit を繰り返しながら多数のファイルを編集する
→ 修正前: 最大1,000ファイル分を丸ごとメモリにピン留め
→ 修正後: キャッシュ合計サイズが16MBを超えないよう管理
```

## 実践例

### 大規模モノレポで多数の大きめファイルを連続編集する

多数のパッケージにまたがるモノレポで、比較的サイズの大きい生成ファイルや設定ファイルを含む数百〜千件規模のファイルを次々と読み書きするような大規模リファクタリングでは、以前はキャッシュがファイル内容を際限なく（最大1,000ファイル分）保持し続け、長時間セッションでメモリ使用量が増加する要因になっていました。サイズ上限方式への変更により、メモリ使用量が安定します。

### メモリ制約のある環境でのCI実行

コンテナやCIランナーなどメモリに制約がある環境でClaude Codeを使って多数のファイルを編集するタスクを実行する場合、キャッシュのメモリ使用量が予測しやすくなったことで、OOM（メモリ不足によるプロセス終了）のリスクを減らせます。

## 注意点

- ユーザー側での設定変更は不要です。キャッシュの挙動は自動的に管理されます。
- 効果が体感しやすいのは、サイズの大きいファイルを多数編集する長時間・大規模なセッションです。
- キャッシュがサイズ上限に達すると古いエントリから追い出される想定であり、編集そのものの正確性（読み込んだ内容との整合性チェック）には影響しません。
