## 原文（日本語訳）

MCPの再認証実行時、新しいサインインが成功する前に既存の有効な認証情報を失効させてしまう問題を修正しました。また、バックグラウンドセッションにおける再接続時の「認証が必要」メッセージが、実行できないコマンドを案内してしまう問題も修正しました。

## 原文（英語）

Fixed MCP re-authenticate revoking working credentials before the new sign-in succeeds, and the reconnect needs-auth message in background sessions pointing at an unusable command

## 概要

Claude CodeでMCPサーバーの再認証（re-authenticate）を行う際、従来は新しいサインインフローが完了する前に、既存の（まだ有効な）認証情報を先に失効させてしまうことがありました。これにより、新しい認証がタイムアウトや失敗などで完了しなかった場合に、それまで正常に使えていたMCPサーバーとの接続まで失われてしまうという問題がありました。あわせて、バックグラウンドで実行中のセッションがMCPサーバーの再接続で認証エラーに遭遇した際に表示される案内メッセージが、そのセッションでは実行できないコマンドを案内してしまう不具合も修正されています。

## 基本的な使い方

この修正はバグ修正のため特別な設定は不要ですが、MCPサーバーの再認証は`/mcp`コマンドから行います。

```
> /mcp
```

対象のサーバーを選択し、再認証（reauthenticate）を実行すると、新しいサインインが成功した時点で初めて古い認証情報が置き換えられるようになりました。

## 実践例

### MCPサーバーを再認証する

```
> /mcp
```

一覧から認証が必要なサーバー、または再認証したいサーバーを選択します。認証フローの途中でブラウザでの承認をキャンセルしたり失敗させたりしても、修正後は元々有効だった認証情報がそのまま維持されるため、MCPサーバーへの接続が失われることはありません。

### CLIから認証状態を確認する

```bash
claude mcp list
```

各MCPサーバーの接続状態と認証状態を一覧で確認できます。

### バックグラウンドセッションでの再接続

バックグラウンドで動作しているセッション（`claude --background`や自動化されたルーティンなど）がMCPサーバーへの再接続時に認証エラーを検知した場合、以前は対話セッションでしか実行できないコマンドを案内するメッセージが表示されることがありました。修正後は、バックグラウンドセッションの文脈に応じた適切な案内（例えば対話セッション側で`/mcp`を実行するようにという案内）が表示されます。

## 注意点

- 再認証中にサインインが失敗した場合、修正後は元の認証情報が維持されるため、必要であれば再度`/mcp`から再認証を試みてください。
- バックグラウンドセッション自体からは対話的な認証操作はできません。認証が必要な場合は、対応する対話型セッションまたは`claude mcp`のCLIコマンドから認証を行う必要があります。
- OAuth形式で認証するMCPサーバーでは、トークンの有効期限切れと再認証失敗は別の問題です。`/mcp`で表示されるステータスを確認し、状況に応じて対処してください。
