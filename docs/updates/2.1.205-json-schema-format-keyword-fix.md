## 原文（日本語訳）

`--json-schema` が、スキーマが不正な場合に何もエラーを出さずに非構造化出力を返してしまう問題と、`format` キーワードを使ったスキーマが拒否されてしまう問題を修正した。

## 原文（英語）

Fixed `--json-schema` silently producing unstructured output when the schema was invalid, and schemas using the `format` keyword being rejected

## 概要

`--json-schema` は、Claude Codeの出力をJSON Schemaで指定した形に強制する（constrained decodingにより出力トークン自体を制約する）CLIフラグで、`--output-format json` と組み合わせて使うことで結果に `structured_output` フィールドが含まれるようになります。今回の修正前は、渡したスキーマ自体が不正な場合にエラーにならず、代わりに構造化されていない通常のテキスト出力へ静かにフォールバックしてしまい、パイプライン側で気づかないまま処理が進む危険がありました。また `date-time` や `email` などの `format` キーワードを含むスキーマを渡すと、有効なJSON Schemaであるにもかかわらず拒否されるバグもあり、今回両方が修正されています。

## 基本的な使い方

```bash
claude -p "3つの国とその首都を挙げて" \
  --output-format json \
  --json-schema countries.schema.json
```

`countries.schema.json` の例:

```json
{
  "type": "object",
  "properties": {
    "countries": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "name": { "type": "string" },
          "capital": { "type": "string" }
        },
        "required": ["name", "capital"]
      }
    }
  },
  "required": ["countries"]
}
```

`--output-format json` の結果に含まれる `structured_output` フィールドに、スキーマへ準拠したJSONが入ります。

## 実践例

### `format` キーワードを使ったスキーマで日時・メールを検証する

修正前は `format: "date-time"` や `format: "email"` を含むスキーマがそもそも受け付けられませんでしたが、修正後はこうしたスキーマも正しく処理されるようになりました。

```json
{
  "type": "object",
  "properties": {
    "reportedAt": {
      "type": "string",
      "format": "date-time"
    },
    "contactEmail": {
      "type": "string",
      "format": "email"
    },
    "summary": { "type": "string" }
  },
  "required": ["reportedAt", "contactEmail", "summary"]
}
```

```bash
claude -p "インシデントレポートの要約を作って" \
  --output-format json \
  --json-schema incident.schema.json
```

### 不正なスキーマを渡した際にエラーで気づけるようにする

CI/CDパイプラインなどでスキーマファイルを自動生成している場合、タイポや構文ミスで不正なJSON Schemaを渡してしまうことがあります。修正前はこのようなケースで黙って非構造化テキストが返り、後続のパース処理が失敗する原因が分かりにくくなっていました。

```bash
# schema.json 内で "type" の値が不正（配列でなければならない箇所が文字列など）
claude -p "レビュー結果を出力して" \
  --output-format json \
  --json-schema schema.json
```

修正後は、不正なスキーマを渡した場合に静かなフォールバックではなく、問題が明確になる形で扱われるため、パイプライン側でスキーマ不備を早期に検知しやすくなります。スキーマを変更した際は、まず単体のコマンドとして実行し `structured_output` フィールドが期待通り含まれているかを確認する運用にしておくと安全です。

## 注意点

- `--json-schema` は単体では効果がなく、`--output-format json` と併用して初めて `structured_output` フィールドが結果に含まれます。
- サポートされるJSON Schemaの機能は、object・array・string・number・boolean・nullなどの基本型、enum、const、required、ネストしたオブジェクト、`$ref` による定義参照など標準的な範囲です。今回の修正で `format` キーワード（`date-time`、`email` など）も利用できるようになりました。
- constrained decodingによって出力トークン自体が制約されるため、スキーマに準拠しない出力が生成されることはありませんが、スキーマ自体が複雑になりすぎるとClaudeが期待した内容を出力しづらくなる場合があります。
- Agent SDKを使う場合は、CLIの `--json-schema` ではなく `query()` の `outputFormat`（TypeScript）/ `output_format`（Python）オプションを使い、Zod（TypeScript）やPydantic（Python）で型安全にスキーマを定義できます。
