## 原文（日本語訳）

`/commit-push-pr` が、`origin` に加えて、リポジトリで設定されているpush先リモート（`remote.pushDefault`、または設定されているリモートが1つだけの場合はそのリモート）への `git push` も自動的に許可するようになった。

## 原文（英語）

`/commit-push-pr` now auto-allows `git push` to the repo's configured push remote (`remote.pushDefault`, or the sole remote when only one is configured) in addition to `origin`

## 概要

`/commit-push-pr` はコミット・プッシュ・PR作成を一気に行う便利なコマンドですが、これまでは `git push` の自動許可対象が `origin` に限定されていました。今回のアップデートで、Gitの `remote.pushDefault` 設定や、リモートが1つしか登録されていない場合のそのリモートに対しても自動的にpushが許可されるようになり、`origin` 以外の名前でリモートを運用しているリポジトリでも確認ダイアログなしでスムーズに使えるようになりました。

## 基本的な使い方

いつも通りセッション内で以下を実行するだけです。

```text
/commit-push-pr
```

`origin` という名前ではないリモート（例えば `upstream` や `fork` など）が push 先として設定されている場合でも、Claudeが許可を求めることなく `git push` を実行します。

## 実践例

### `remote.pushDefault` を設定しているリポジトリ

複数リモートを使い分けているプロジェクトで、以下のようにpush先を明示的に設定している場合を考えます。

```bash
git config remote.pushDefault upstream
```

この状態で `/commit-push-pr` を実行すると、`upstream` への `git push` が自動的に許可され、余計な確認プロンプトが表示されません。

### リモートが1つしか登録されていないクローン

社内ミラーなど、`origin` 以外の名前（例: `internal`）でリモートを1つだけ登録しているリポジトリでも、`/commit-push-pr` はそのリモートを自動検出し、pushの許可を求めることなく処理を進めます。

```text
/commit-push-pr
```

## 注意点

- 複数のリモートが登録されており、かつ `remote.pushDefault` が設定されていない場合は、従来通り `origin` 以外への push は自動許可の対象外です。
- 自動許可はあくまで「確認ダイアログを省略する」という挙動であり、pushの実行自体は通常の権限設定（`allow`/`ask`/`deny`）の影響を受けます。
- フォーク運用など、push先とPR作成先のリモートが異なる場合は、コマンドの挙動を事前に確認しておくと安心です。
