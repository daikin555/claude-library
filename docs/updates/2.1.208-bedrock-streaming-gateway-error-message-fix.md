## 原文（日本語訳）

ゲートウェイがレスポンスを変換した場合に、Bedrockのストリーミングリクエストが誤解を招く「Truncated event message received（イベントメッセージが切り詰められて受信された）」というエラーで失敗していた問題を修正。エラーは、受信したcontent-typeの名前とプロキシを指し示すようになった。

## 原文（英語）

Fixed Bedrock streaming requests failing with a misleading "Truncated event message received" when a gateway transforms the response — the error now names the content-type and points at the proxy

## 概要

Amazon BedrockはInvokeModelWithResponseStreamのレスポンスを `application/vnd.amazon.eventstream` というバイナリのイベントストリーム形式で返します。Claude CodeとBedrockの間にAPI GatewayやLambdaなどのゲートウェイ・プロキシを挟んでいる構成では、このバイナリストリームがserver-sent events形式（`text/event-stream` など）に変換されてしまうことがあり、Claude Codeはデコードできないボディを黙って読もうとして失敗していました。これまではその際、原因が分かりにくい「Truncated event message received」というエラーだけが表示されていましたが、今回のアップデートでエラーメッセージが実際に受信したcontent-typeの種類を明示し、ゲートウェイ側の設定が原因であることを直接示すように改善されました。

## 基本的な使い方

Amazon Bedrock経由でClaude Codeを使う設定自体は変わりません。

```bash
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION=us-east-1
```

ゲートウェイやプロキシ経由でストリーミングレスポンスが変換されている場合、修正後は次のような形式でエラーが表示されます。

```text
Bedrock streaming response has content-type ... (expected application/vnd.amazon.eventstream)
```

このメッセージからゲートウェイ側でストリーミングレスポンスが変換されていることが直接わかります。

## 実践例

### API GatewayとLambdaでBedrockをプロキシしている場合

社内のネットワークポリシー上、AWS API GatewayとLambdaを介してBedrockへのリクエストをプロキシしている構成では、Lambda側の実装によってバイナリのイベントストリームがserver-sent events形式に再エンコードされてしまうことがあります。

```bash
export CLAUDE_CODE_USE_BEDROCK=1
export ANTHROPIC_BEDROCK_BASE_URL=https://your-gateway.example.com
```

以前はこの状態で「Truncated event message received」という漠然としたエラーしか出ず、原因の切り分けに時間がかかっていましたが、修正後は `text/event-stream` のような実際のcontent-typeが表示されるため、ゲートウェイ側の変換が原因だとすぐに特定できます。

### ゲートウェイの修正がすぐにできない場合の回避策

ゲートウェイのヘッダーだけが書き換わっていて、バイナリボディ自体はそのまま素通しされている場合は、一時的にcontent-typeのチェックを無効化して動作を継続できます。

```bash
export CLAUDE_CODE_DISABLE_BEDROCK_CONTENT_TYPE_GUARD=1
```

ただし、ボディ自体が変換されている場合にこの環境変数でチェックを無効化すると、今度は元の「Truncated event message received」エラーが再発するため、恒久対応としてはゲートウェイ側で `InvokeModelWithResponseStream` のレスポンスボディとその `Content-Type` ヘッダーを無変換で通過させるよう設定することが推奨されます。

## 注意点

- このエラーは、Bedrockが返すバイナリのイベントストリーム（`application/vnd.amazon.eventstream`）とは異なるcontent-typeを受け取った場合にのみ発生します。ゲートウェイやプロキシを挟んでいない直接接続では通常発生しません。
- `CLAUDE_CODE_DISABLE_BEDROCK_CONTENT_TYPE_GUARD=1` はあくまで一時的な回避策であり、ボディ自体が変換されているケースでは根本解決になりません。
- この改善はClaude Code v2.1.208以降が対象です。それより前のバージョンでは、同じ設定ミスが「Truncated event message received」という誤解を招くエラーとしてのみ表示されていました。
- Claude CodeはBedrockの `InvokeModelWithResponseStream`（Invoke API）を使用しており、Converse APIはサポートしていません。
