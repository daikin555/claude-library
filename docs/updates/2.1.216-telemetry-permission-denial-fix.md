## 原文（日本語訳）

権限拒否に関するテレメトリの誤集計を修正しました。権限プロンプトのリクエストが失敗した場合、ユーザーによる拒否としてカウントされなくなり、ユーザーによる中断は拒否ではなく「ユーザーによる中止（abort）」として報告されるようになりました。

## 原文（英語）

Fixed telemetry misreporting permission denials: failed permission-prompt requests no longer count as user rejections, and user interrupts are now reported as user aborts instead of rejections

## 概要

Claude CodeはOpenTelemetry経由で権限プロンプトの決定内容（`decision_source` など）をテレメトリとして送信できますが、これまでは「権限プロンプトのリクエスト自体が失敗したケース」と「ユーザーが本当に拒否を選んだケース」が区別されずどちらも拒否として集計され、また「ユーザーがEscなどで処理を中断したケース」も拒否として扱われていました。この修正により、リクエスト失敗はユーザーの拒否としてカウントされなくなり、ユーザーの中断は新たに「ユーザーによる中止（abort）」という区分で報告されるようになります。これにより、監視ダッシュボードやセキュリティ検知ルールが実態に即したより正確な集計を行えるようになります。

## 基本的な使い方

この修正は自動的に適用される挙動であり、追加の設定は不要です。テレメトリを有効化している場合、既存の設定のままアップグレードするだけで、より正確な `decision_source` が送信されるようになります。

```bash
export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_METRICS_EXPORTER=otlp
export OTEL_LOGS_EXPORTER=otlp
```

## 実践例

### 権限拒否イベントをダッシュボードで正しく分類する

`decision_source` フィールドを使って、Grafanaやログ集計基盤上で「ユーザーが明示的に拒否した操作」と「ユーザーが処理を中断した操作」を分けて可視化できます。

```
decision_source = "user_reject"  → ユーザーが明示的に拒否
decision_source = "user_abort"   → ユーザーが中断（Escなど）
```

修正前はこの2つが区別されず、中断もすべて拒否としてカウントされていたため、実際の拒否率が過大に見積もられていました。

### 権限プロンプトの通信エラーをノイズとして除外する

ネットワークの瞬断などで権限プロンプトのリクエスト自体が失敗した場合、修正前はこれもユーザー拒否として計上されていましたが、修正後はカウント対象から外れます。これにより、セキュリティ監視ルールで「特定コマンドが頻繁に拒否されている」といったアラートを組む際の誤検知が減ります。

```
# 検知ルール例（疑似コード）
alert if count(decision_source == "user_reject" AND tool == "Bash") > threshold
```

### インシデント調査時にユーザーの実際の意図を追いやすくする

セッションログを後から調査する際、「Claudeの提案を拒否した」のか「単に作業を中断した」のかが `decision_source` から明確に判別できるようになり、監査やレビューの精度が向上します。

## 注意点

- この修正はテレメトリの集計・分類ロジックに関するものであり、Claude Code本体の権限判断の挙動自体（許可/拒否の判定）は変わりません
- 既存のダッシュボードやアラートルールで `user_reject` のみを拒否として集計していた場合、修正後は数値が変動する可能性があります。必要に応じて `user_abort` も含めて集計するようクエリを見直してください
- テレメトリの詳細な `decision_source` の値（`config` / `hook` / `user_permanent` / `user_temporary` / `user_reject` / `user_abort` など）は今後のバージョンで追加・変更される可能性があります
