## 原文（日本語訳）

`.claude`がシンボリックリンクになっている場合、ワークフローの保存やスケジュールタスクの書き込みがそのシンボリックリンクをたどってしまい、書き込み先がプロジェクトの外にリダイレクトされる可能性がある問題を修正しました。

## 原文（英語）

Fixed workflow saves and scheduled-task writes following a symlink at `.claude`, which could redirect writes outside the project

## 概要

Claude Codeはプロジェクトルートの`.claude`ディレクトリにワークフローやスケジュールタスクの定義ファイルを書き込みます。もし`.claude`自体がシンボリックリンクとして構成されていた場合、従来の実装ではそのリンクをそのままたどって書き込みを行っており、意図しない場所（プロジェクトディレクトリの外）にファイルが作成・上書きされてしまう可能性がありました。これはシンボリックリンクを悪用することでプロジェクト外への書き込みを誘発できる、セキュリティ上のリスクとなり得る問題でした。今回の修正により、`.claude`がシンボリックリンクであってもワークフロー保存やスケジュールタスクの書き込みがプロジェクト範囲外にリダイレクトされないようになりました。

## 基本的な使い方

この修正はセキュリティ関連の内部挙動の改善であり、通常の利用でユーザーが意識する設定変更は不要です。ワークフローの保存やスケジュールタスクの作成はこれまでどおり行えます。

```
> /schedule
```

## 実践例

### スケジュールタスクを作成する

```
> /schedule 毎朝9時に依存パッケージの更新確認を実行して
```

このコマンドで生成されるタスク定義ファイルは、`.claude`ディレクトリが通常のディレクトリであってもシンボリックリンクであっても、常にプロジェクトの管理下にある想定のパスに書き込まれます。

### `.claude`ディレクトリの状態を確認する

プロジェクト内で`.claude`が誤ってシンボリックリンクになっていないかを確認したい場合は、以下のように確認できます。

```bash
ls -la .claude
```

もし`.claude -> /some/other/path`のようなシンボリックリンクが表示された場合、意図した構成かどうかを確認してください。

### ワークフローを保存する

```
> /workflow save my-workflow
```

保存先はプロジェクト内の`.claude/workflows/`配下に限定され、シンボリックリンク経由でプロジェクト外の任意の場所へ書き込まれることはありません。

## 注意点

- 意図的に`.claude`をシンボリックリンクとして運用している構成（モノレポでの共有設定など）がある場合、書き込み先の解決ロジックが変更されている可能性があるため、アップデート後に動作を確認することを推奨します。
- この修正はパストラバーサル/シンボリックリンク経由の書き込みリダイレクトを防ぐセキュリティ強化であり、通常の正当な用途への影響はありません。
- 過去のバージョンで既に意図しない場所に書き込まれたファイルがある場合、それらは自動的にはクリーンアップされません。必要に応じて手動で確認・削除してください。
