## 原文（日本語訳）

多数の権限deny/askルールを持つセッションで発生していた、ターンごとの数秒単位の速度低下を修正した。ルールマッチャーは一度だけコンパイルされ、キャッシュされるようになった。

## 原文（英語）

Fixed multi-second per-turn slowdowns in sessions with many permission deny/ask rules — rule matchers are now compiled once and cached

## 概要

Claude Codeは `settings.json` の `permissions` に定義された `allow`/`ask`/`deny` ルールを、ツール呼び出しのたびに評価してアクセス可否を判断します。これまでは、ルールの数が多い環境ではツール呼び出しごとにルールマッチャーを再コンパイルしており、企業ポリシーなどで数十〜数百件のdeny/askルールを設定しているセッションではターンごとに数秒単位の遅延が発生していました。今回の修正でルールマッチャーは初回に一度だけコンパイルされてキャッシュされるようになり、ルール数が多い環境でも応答速度が改善しました。

## 基本的な使い方

これはユーザーが明示的に呼び出す機能ではなく、内部的なパフォーマンス改善です。特別な操作は不要で、アップデート後は自動的に適用されます。

```json
// .claude/settings.json
{
  "permissions": {
    "deny": ["Bash(curl *)", "Read(./secrets/**)", "..."],
    "ask": ["Bash(git push *)", "..."]
  }
}
```

このような `permissions` 設定を持つプロジェクトで、通常どおりClaude Codeを使うだけで恩恵を受けられます。

## 実践例

### 企業ポリシーで多数のdeny/askルールを運用している組織

セキュリティ要件から `Bash`, `Read`, `Write` などに対して数十件〜数百件のdeny/askルールをチーム全体の `.claude/settings.json` に設定している場合、以前はツール呼び出しのたびにルール一致判定のコンパイルコストがかかり、ターンごとに数秒の遅延として体感されていました。修正後はマッチャーが初回にキャッシュされるため、ルール数が多くても遅延が発生しにくくなります。

```json
{
  "permissions": {
    "deny": [
      "Bash(aws *)", "Bash(curl *)", "Bash(rm -rf *)",
      "Read(./.env)", "Read(./secrets/**)"
    ],
    "ask": [
      "Bash(git push *)", "Bash(npm publish *)"
    ]
  }
}
```

### 長時間セッションでの累積的な遅延の解消

長時間にわたり多くのツール呼び出しを繰り返すエージェントセッションでは、ルール数が多いほど遅延が積み重なりやすい状況でした。マッチャーのキャッシュ化により、セッションを通じて安定した応答速度を維持しやすくなります。

## 注意点

- ユーザー側で設定変更は不要です。既存の `permissions` 設定はそのまま動作します。
- 効果が体感しやすいのは、deny/askルールを多数（数十件以上）定義している大規模プロジェクトやエンタープライズ環境です。
- ルールの評価順序（deny → ask → allow）や挙動自体は変更されておらず、あくまで内部処理の高速化です。
