## 原文（日本語訳）

Bedrock: エグレス（外向き通信）が制限されたネットワークで `awsCredentialExport` ヘルパーを使用した際に、起動時に数分間ハングする問題を修正した

## 原文（英語）

Bedrock: fixed a multi-minute startup hang when using an `awsCredentialExport` helper on networks with restricted egress

## 概要

Amazon Bedrock経由でClaude Codeを利用する際、デフォルトのAWS認証情報プロバイダーチェーンでは対応できないクロスアカウント認証などが必要な場合に、`awsCredentialExport` 設定でカスタムの認証情報取得コマンドを指定できます。このヘルパーはセッション開始時や認証情報の再読み込みのたびに呼び出されますが、外向き通信（エグレス）が制限されたネットワーク環境では、内部的なネットワークアクセスの試行がタイムアウトするまで待たされ、結果としてClaude Codeの起動そのものが数分間ハングしてしまう問題がありました。今回の修正でこの待機処理が改善され、制限されたネットワーク環境でも起動が速やかに完了するようになりました。

## 基本的な使い方

`settings.json` で `awsCredentialExport` を設定してBedrockを利用している場合、特別な追加設定は不要で、アップデート後のClaude Codeを起動するだけで修正が適用されます。

```json
{
  "awsCredentialExport": "/path/to/get-bedrock-credentials.sh"
}
```

```bash
export CLAUDE_CODE_USE_BEDROCK=1
claude
```

## 実践例

### 社内プロキシ経由でしか外部通信ができない環境

社内ネットワークのセキュリティポリシーにより一般的なインターネットへのエグレスが制限されており、Bedrockへのアクセスや認証情報取得スクリプトの実行に専用の内部エンドポイントしか使えないような環境です。以前はこうした制限環境で `claude` コマンドの起動自体が数分間応答しなくなることがありましたが、修正後は起動時間が大幅に改善されます。

### クロスアカウントのIAMロールを引き受けて認証情報を発行するスクリプトを使う場合

`aws configure export-credentials` のようなコマンド、またはそれと同等のJSON形式で `AccessKeyId` / `SecretAccessKey` / `SessionToken` を返す独自スクリプトを `awsCredentialExport` に指定し、クロスアカウントロールを引き受けて一時的な認証情報を発行しているケースです。ネットワーク制限がある環境でもこの仕組みを安心して使えるようになりました。

### CI/CDやサンドボックス環境でBedrockを使う場合

エグレスが厳しく制限されたCIランナーやサンドボックス環境でClaude Code + Bedrockを組み込んでいる場合、以前は起動の遅延がパイプライン全体の実行時間に影響することがありましたが、この修正によって起動遅延のリスクが軽減されます。

## 注意点

- この修正は起動時の待機時間に関するものであり、`awsCredentialExport` の設定方法や返却すべきJSON形式自体に変更はありません。
- 認証情報の有効期限（`Expiration`）を返す場合、Claude Codeはその5分前まで認証情報をキャッシュする仕様であり、この挙動自体は今回の修正の対象ではありません。
- 依然としてエグレスが完全に遮断されている場合、ヘルパースクリプト自体の実行やBedrockへの接続には別途ネットワーク経路の確保が必要です。
- 公式ドキュメントには本修正の詳細（内部的な待機処理の変更点）についての明記が本記事作成時点では見当たらなかったため、changelogの記載内容をもとに記述しています。
