## 原文（日本語訳）

置き換え済みのファイル履歴バックアップを整理することで、セッションのトランスクリプトサイズを削減し（編集の多いセッションでは最大79倍）、チェックポイントのディスク使用量にも上限を設けた。

## 原文（英語）

Reduced session transcript size (up to 79x in edit-heavy sessions) and bounded checkpoint disk usage by pruning superseded file-history backups

## 概要

Claude Codeは `/rewind` によるチェックポイント機能のために、ファイルを編集するたびにその時点のファイル内容をバックアップとして保持しています。これまでは同じファイルに対する過去のバックアップがすべて残り続けるため、1つのファイルを何度も編集するセッションではセッショントランスクリプト（会話ログ）やディスク使用量が肥大化しがちでした。今回の変更では、後続の編集によって不要になった（置き換え済みの）過去のバックアップを整理するようになり、編集の多いセッションではトランスクリプトサイズが最大79倍削減され、チェックポイント用のディスク使用量にも上限が設けられました。

## 基本的な使い方

これはユーザーが明示的に呼び出す機能ではなく、内部的なストレージ管理の改善です。特別な操作は不要で、`/rewind` や自動チェックポイント機能を使う中で自動的に適用されます。

```text
Claudeが同じファイルを何度も編集する
→ 修正前: すべての中間バックアップがトランスクリプトに蓄積
→ 修正後: 不要になった中間バックアップは整理され、
          必要なチェックポイント復元性は保ったままサイズが削減される
```

## 実践例

### 同じファイルを繰り返し試行錯誤しながら編集するセッション

1つのコンポーネントやテストファイルに対してClaudeが何度も修正・再修正を繰り返すようなセッション（例えばテストが通るまで実装を調整し続けるループ）では、以前は編集のたびにファイル全体のバックアップがトランスクリプトに積み上がり、セッションファイルが肥大化してセッションの再開や共有が重くなることがありました。修正後は置き換え済みのバックアップが整理されるため、長時間の試行錯誤セッションでもトランスクリプトサイズが抑えられます。

### `/rewind` を多用する開発フローでのディスク使用量の安定化

こまめに `/rewind` でチェックポイントを遡りながら開発を進めるスタイルのユーザーでも、チェックポイント用のディスク使用量に上限が設けられたことで、ディスクを圧迫し続ける心配が少なくなります。

```text
/rewind
```

でチェックポイント一覧を開き、必要な時点まで戻る操作自体は従来通り利用できます。

## 注意点

- 整理されるのは「後続の編集によって置き換えられ、もはや復元に必要のないバックアップ」のみで、`/rewind` で実際に必要な直近の復元ポイントは引き続き保持されます。
- ユーザー側での設定変更は不要です。
- 効果が体感しやすいのは、同一ファイルへの編集回数が多い（編集の多い）セッションです。編集回数が少ないセッションでは差は小さくなります。
- チェックポイントはあくまでセッション内の「ローカルなundo」であり、恒久的な変更履歴の管理にはGitなどのバージョン管理システムを引き続き使用することが推奨されています。
