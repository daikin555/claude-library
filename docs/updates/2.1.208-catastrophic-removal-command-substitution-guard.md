## 原文（日本語訳）

`rm -rf ~`のような破壊的（catastrophic）な削除を含むコマンドで、`$(…)`やバッククォート、`<(…)`のようなコマンド置換を含む場合も、`--dangerously-skip-permissions`とautoモードで確認プロンプトが表示されるようになり、通常形式のコマンドと同じ扱いになった。

## 原文（英語）

Catastrophic removals (e.g. `rm -rf ~`) in commands containing `$(…)`/backticks/`<(…)` now prompt in `--dangerously-skip-permissions` and auto mode, matching the plain form

## 概要

Claude Codeには、`rm -rf /`や`rm -rf ~`のようにファイルシステムのルートやホームディレクトリを対象にした破壊的な削除コマンドに対して、モデルの誤りに対するサーキットブレーカーとして常に確認プロンプトを出す仕組みがあります。これは`bypassPermissions`（`--dangerously-skip-permissions`）モードやautoモードでも例外なく発動します。しかし今回の修正以前は、同じ削除コマンドが`$(…)`やバッククォート、`<(…)`といったコマンド置換・プロセス置換の中に埋め込まれていると、このサーキットブレーカーを素通りしてautoモードの分類器（classifier）の判断だけに委ねられてしまうケースがありました。v2.1.208でこの抜け道が塞がれ、置換の内側にあっても外側にあっても、破壊的な削除は通常形式と同じく必ず確認を求めるようになりました。

## 基本的な使い方

このサーキットブレーカーはユーザーが明示的に呼び出す機能ではなく、`bypassPermissions`モードやautoモードでClaude Codeが破壊的なコマンドを実行しようとした際に自動的に働きます。

```bash
claude --dangerously-skip-permissions
```

上記のようにパーミッションチェックを原則スキップするモードで起動していても、次のようなコマンドを実行しようとすると確認プロンプトが表示されます。

```bash
rm -rf ~
```

## 実践例

### コマンド置換の中に危険な削除が埋め込まれているケース

```bash
echo "$(rm -rf ~)"
```

このように`$(…)`の内側に破壊的な削除コマンドが隠れている場合、v2.1.208より前はautoモードの分類器の判断のみで実行されてしまう可能性がありましたが、修正後は通常の`rm -rf ~`と同様に確認プロンプトが表示されるようになりました。

### バッククォートやプロセス置換と組み合わされているケース

```bash
diff <(rm -rf ~) /dev/null
```

`<(…)`のようなプロセス置換や、バッククォート`` `rm -rf ~` ``のような古い形式のコマンド置換と組み合わせて、破壊的な削除コマンドが同じコマンド文字列のどこかに含まれている場合も、同様に確認が求められるようになります。これにより、悪意のあるプロンプトインジェクションや意図しないコマンド生成によって`--dangerously-skip-permissions`や`auto`モードが破壊的な操作を素通りしてしまうリスクが減ります。

## 注意点

- サーキットブレーカーが対象とするのは、ファイルシステムのルートやホームディレクトリを狙う破壊的な削除（例: `rm -rf /`、`rm -rf ~`）です。それ以外の一般的な削除コマンドはautoモードの分類器による判断や、通常のパーミッションルールに従います。
- `bypassPermissions`（`--dangerously-skip-permissions`）モードは、コンテナやVMなど隔離された環境でのみ使用することが推奨されています。ホストマシン上での常用は避けてください。
- 明示的な`ask`ルールは、`bypassPermissions`モードを含むあらゆるモードで確認プロンプトを強制します。
- autoモードの分類器はユーザーのメッセージ・ツール呼び出し・CLAUDE.mdの内容は参照しますが、ツールの実行結果はサニタイズされて渡されないため、ツール結果に埋め込まれた悪意あるコンテンツから直接操作されることはありません。
