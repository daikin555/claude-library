## 原文（日本語訳）

長時間のセッションで発生していた複数のメモリリークを修正した。内容は、MCP stdioサーバーのstderrがサーバーごとに最大64MBまで蓄積してしまう問題、LSPドキュメントが無期限に開いたままになる問題（現在は50ドキュメント上限のLRUキャッシュ方式に変更）、バックグラウンド化後も非同期フックの出力が保持され続ける問題、そしてヘッドレス/SDKセッションで大きなツール結果のペイロードによりメモリが際限なく増加する問題。

## 原文（英語）

Fixed several memory leaks in long sessions: MCP stdio server stderr accumulating up to 64 MB per server, LSP documents staying open indefinitely (now LRU with 50-doc cap), async hook output retained after backgrounding, and unbounded growth in headless/SDK sessions from large tool-result payloads

## 概要

Claude Codeを長時間、あるいは自動化パイプラインの中で連続稼働させる場合、内部のさまざまなサブシステムに存在していたメモリリークが積み重なり、メモリ使用量が徐々に増加していく問題がありました。今回のアップデートでは、[MCP](https://code.claude.com/docs/en/mcp) stdioサーバーの標準エラー出力（stderr）がサーバー1つあたり最大64MBまで蓄積してしまう問題、LSPで開いたドキュメントが閉じられず無期限に保持されていた問題（50ドキュメント上限のLRUキャッシュへ変更）、フックが非同期でバックグラウンド化した後もその出力を保持し続けていた問題、そして[ヘッドレスモード](https://code.claude.com/docs/en/headless)やSDKセッションで大きなツール結果のペイロードによりメモリが際限なく増加していた問題、という4つのメモリリークがまとめて修正されました。

## 基本的な使い方

これらはすべて内部実装の修正であり、ユーザー側で特別な操作や設定変更は不要です。Claude Codeを最新版（v2.1.208以降）に更新するだけで適用されます。

```bash
claude update
# または各パッケージマネージャでのアップデート
```

## 実践例

### MCPサーバーを複数接続した長時間セッション

複数のMCP stdioサーバーを接続した状態で1日を通してセッションを継続するような使い方をしていると、修正前は各サーバーのstderr出力が最大64MBまで蓄積し、メモリ使用量を圧迫していました。

```json
// .mcp.json の例（複数のstdioサーバーを接続）
{
  "mcpServers": {
    "server-a": { "command": "node", "args": ["./server-a.js"] },
    "server-b": { "command": "node", "args": ["./server-b.js"] }
  }
}
```

修正後は、各サーバーのstderr蓄積が抑えられ、長時間の接続でもメモリ使用量が無制限に増え続けることがなくなりました。

### LSPを有効にしたまま多数のファイルを編集する長時間コーディングセッション

大規模なリファクタリングなどで、LSPを有効にしたまま次々と多数のファイルを開いて編集していくと、修正前はLSPドキュメントが無期限に開いたままメモリに保持され続けていました。修正後は50ドキュメントを上限としたLRU（Least Recently Used）キャッシュ方式となり、直近使用していないドキュメントは自動的に解放されます。

```text
# 数百ファイルにまたがるリファクタリングを1つの長いセッションで実施
# LSPドキュメントは直近使用した50件のみメモリに保持される
```

### ヘッドレス/SDKで大きなツール結果を返す処理を長時間ループ実行するケース

`claude -p` によるヘッドレス実行や、Agent SDKを使った自動化パイプラインで、大きな出力を返すツール呼び出しを繰り返すループを長時間動かしていると、修正前はツール結果のペイロードによってメモリが際限なく増加していく問題がありました。

```bash
# CIパイプラインなどでヘッドレスモードを繰り返し実行する例
claude -p "大きなログファイルを解析してレポートを作成して" --output-format json
```

修正後は、こうした大きなツール結果を扱う長時間の自動化処理でもメモリ増加が抑えられます。

## 注意点

- いずれも内部的なメモリ管理の改善であり、設定変更やコード変更は不要です。
- LSPドキュメントのLRUキャッシュ化（50ドキュメント上限）により、非常に多数のファイルを同時に行き来するような極端なケースでは、直近使用していないドキュメントが再度開き直されることがありますが、メモリ使用量とのトレードオフとして導入されています。
- 特にMCPサーバーを多数接続する構成、LSPを有効にした長時間の大規模編集、CI/自動化パイプラインでのヘッドレス/SDK連続実行など、長時間・高頻度にセッションを使う運用ほど今回の修正の恩恵が大きくなります。
