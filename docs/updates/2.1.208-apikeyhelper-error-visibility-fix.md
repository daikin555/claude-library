## 原文（日本語訳）

`apiKeyHelper` スクリプトの失敗が、約10回の無言リトライの末に汎用的な401エラーの裏に隠れてしまう問題を修正。スクリプト自身のエラーが3回の試行以内に表示されるようになった。

## 原文（英語）

Fixed `apiKeyHelper` script failures being hidden behind a generic 401 after ~10 silent retries; the script's own error is now shown within 3 attempts

## 概要

`apiKeyHelper` は、`settings.json` に設定したスクリプトを実行してAPIキー（認証値）を動的に取得する仕組みです。このスクリプトがエラーで終了したり、タイムアウトしたり、標準出力に何も出力しなかったりすると、有効なキーが得られないままプレースホルダーの認証情報でリクエストが送られ、APIから401エラーが返っていました。これまでは、この401エラーが出るまでにリトライ予算をすべて使い切る形で約10回も無言でリトライが繰り返されており、本来の原因であるスクリプト自体のエラー内容がユーザーに見えませんでした。今回のアップデートで、スクリプトを再実行しつつ最大2回まで追加でリトライした時点（=最大3回の試行以内）で、スクリプト自身のエラー内容を明示するようになりました。

## 基本的な使い方

`apiKeyHelper` は `settings.json` でスクリプトのパスを指定して設定します。

```json
{
  "apiKeyHelper": "~/.claude/anthropic_key_helper.sh"
}
```

このスクリプトが失敗するようになった場合、修正後のClaude Codeは次のようなメッセージを表示します。

```text
Your apiKeyHelper script is failing · This usually means you need to re-authenticate with your provider · Run /status to see the script's error output
```

`/status` を実行すると、スクリプトが実際に出力したエラー内容を確認できます。

## 実践例

### 一時的な認証トークンを発行するスクリプトが期限切れになった場合

社内のSSO基盤やクラウドの一時クレデンシャル発行の仕組みと連携して `apiKeyHelper` スクリプトを組んでいる場合、裏側のトークンが期限切れになるとスクリプトがエラー終了することがあります。

```bash
#!/bin/sh
# ~/.claude/anthropic_key_helper.sh
token=$(get-temp-token.sh) || exit 1
echo "$token"
```

以前はこの失敗が約10回の無言リトライの末に「401エラー」としてしか表示されず、原因の切り分けに時間がかかっていましたが、修正後は3回以内に「apiKeyHelperスクリプトが失敗している」ことと `/status` での確認方法が明示され、すぐに原因調査に取り掛かれます。

### スクリプトの記述ミスや権限不足でクラッシュする場合

新しく `apiKeyHelper` を設定した直後、実行権限が付与されていなかったり、パスの指定を誤っていたりしてスクリプトが即座に失敗するケースもよくあります。

```json
{
  "apiKeyHelper": "/bin/generate_temp_api_key.sh"
}
```

修正後は、こうした設定ミスによる失敗も早い段階で「apiKeyHelperが失敗している」という形で表面化するため、単なる401エラーとして見過ごしてAnthropicアカウント側の問題と誤認することを防げます。

## 注意点

- この修正はエラーの可視性に関するものであり、`apiKeyHelper` の設定方法自体に変更はありません。
- スクリプトは「エラー終了」「タイムアウト」「標準出力への出力なし」のいずれかの場合に失敗と判定されます。
- 失敗時の詳細なエラー出力は `/status` コマンドで確認します。
- この改善はClaude Code v2.1.208以降が対象です。それより前のバージョンでは、スクリプトの失敗はリトライ予算を使い切った末の汎用的な401エラーとしてのみ表示されていました。
