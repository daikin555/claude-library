## 原文（日本語訳）

Windows環境において、worktree内にNTFSジャンクションまたはディレクトリのシンボリックリンクが存在する場合、worktreeを削除すると、そのworktreeの外側にあるファイルまで削除されてしまう不具合を修正した。

## 原文（英語）

Fixed Windows worktree removal deleting files outside the worktree when an NTFS junction or directory symlink existed inside it

## 概要

Claude Codeのworktree機能は、複数のセッションを別々の作業ディレクトリに分離し、並行してタスクを実行できるようにするものです。今回の修正前は、Windows環境でworktreeを削除する際、その中にNTFSジャンクションやディレクトリのシンボリックリンクが含まれていると、削除処理がリンク先の実体をたどってしまい、worktreeの外側にある無関係なファイル（他のプロジェクトやユーザープロファイル配下のフォルダなど）まで再帰的に削除されてしまう危険な不具合がありました。例えばpnpmなどのパッケージマネージャーは`node_modules`内にNTFSジャンクションを作成することがあり、そうした環境でworktreeを削除すると意図せず大規模なデータ消失につながる恐れがありました。本修正により、削除処理はリンク自体のみを削除し、リンク先の実体には手を触れないよう改善されています。

## 基本的な使い方

worktreeの作成・削除自体の操作方法はこれまでと変わりません。Claude Codeに対して自然言語で作業用worktreeの利用を指示するか、CLIから作成・終了を行います。

```text
> このリポジトリでworktreeを使って作業して
```

作業が終わったらworktreeを終了（削除）します。内部的には`EnterWorktree`/`ExitWorktree`に相当する処理が行われ、不要になったworktreeディレクトリがクリーンアップされます。

## 実践例

### pnpmなど、node_modules内にジャンクションを作るツールを使うプロジェクト

WindowsでNode.jsプロジェクトを扱う際、pnpmはディスク容量削減のため`node_modules`配下にNTFSジャンクションを作成することがあります。修正前のバージョンでは、こうしたプロジェクトのworktreeを削除しようとすると、ジャンクションが指す先（グローバルなpnpmストアや別プロジェクトのディレクトリなど）まで巻き込んで削除される恐れがありました。

```text
> このworktreeでの作業が終わったので削除して
```

修正後は、ジャンクション自体（リンクポインタ）だけが削除され、リンク先のファイル・ディレクトリの実体には影響しません。

### 複数プロジェクトを並行してworktreeで管理している場合

複数のworktreeを行き来しながら開発している場合、一方のworktree内に他のディレクトリへのシンボリックリンクを手動で作成しているケースもあります。この修正により、そうした構成でも安全にworktreeの削除・クリーンアップができるようになりました。

```text
> 使い終わったworktreeを一括で整理して
```

## 注意点

- この不具合はWindows特有の問題で、NTFSジャンクションやディレクトリのシンボリックリンクがworktree内部に存在する場合にのみ発生します。macOS/Linuxでのシンボリックリンクの扱いには影響しません。
- 修正前のバージョンを使っている場合、pnpmなどジャンクションを作成するツールと組み合わせたプロジェクトでworktreeを削除すると、意図しないファイル消失につながるリスクがあったため、該当バージョンを使っている場合は速やかにアップデートすることが推奨されます。
- 重要なデータについては、Claude Codeのバージョンにかかわらず、定期的なバックアップやバージョン管理（git）による保護を行っておくことが安全です。
