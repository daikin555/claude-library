## 原文（日本語訳）

初期化に失敗するプラグインのLSPサーバーが、同じファイル拡張子を扱う別プラグインの有効なLSPサーバーの動作を妨げてしまう不具合を修正した。

## 原文（英語）

Fixed a plugin LSP server that fails to initialize preventing a valid LSP server from another plugin handling the same file extension

## 概要

Claude Codeのプラグインは、Language Server Protocol（LSP）サーバーを組み込んで、コード編集時のリアルタイム診断や定義ジャンプ、参照検索などのコードインテリジェンス機能をClaudeに提供できます。これまでは、同じファイル拡張子（例: `.py`や`.go`）を複数のプラグインがそれぞれ別のLSPサーバーで扱おうとした場合、一方のLSPサーバーの初期化に失敗すると、もう一方の正常なLSPサーバーまで巻き込まれて動作しなくなる不具合がありました。今回の修正により、あるプラグインのLSPサーバーが初期化に失敗しても、同じ拡張子を扱う別プラグインの正常なLSPサーバーはそのまま機能するようになりました。

## 基本的な使い方

プラグインでLSPサーバーを使うには、プラグインのルートに `.lsp.json` を置くか、`plugin.json` 内に `lspServers` を直接記述します。

```json
{
  "name": "my-python-lsp",
  "lspServers": {
    "python": {
      "command": "pyright-langserver",
      "args": ["--stdio"],
      "extensionToLanguage": {
        ".py": "python"
      }
    }
  }
}
```

`command` にはLSPバイナリ（PATH上に存在する必要があります）、`extensionToLanguage` には拡張子と言語識別子の対応を指定します。プラグインを有効化すると、対象拡張子のファイルを編集した際にClaudeがこのLSPサーバー経由で診断情報や定義参照を取得できるようになります。

## 実践例

### 複数のLSPプラグインを併用しているプロジェクト

モノレポなどで、公式マーケットプレイスの `typescript-lsp` と、社内独自の `.tsx` 用カスタムLSPプラグインを両方インストールしているようなケースを考えます。今回の修正前は、カスタムLSPプラグイン側のバイナリがPATHに存在しない、あるいは設定ミスで初期化に失敗すると、本来正常に動くはずの `typescript-lsp` まで `.ts`/`.tsx` ファイルで機能しなくなることがありました。修正後は、カスタムLSPプラグインの初期化失敗が独立したエラーとして扱われ、`typescript-lsp` は影響を受けずに診断・定義ジャンプなどを提供し続けます。

```bash
# インストール済みプラグインとLSPの状態を確認
claude plugin list
```

### 一部のプラグインでLSPバイナリが未インストールの場合

チームメンバーの環境によっては、あるプラグインが要求する言語サーバーのバイナリ（例: `rust-analyzer`）だけがインストールされていないことがあります。この場合、そのプラグインのLSPサーバーは `Executable not found in $PATH` のようなエラーになりますが、修正後はこのエラーが該当プラグインの範囲に閉じ、同じ拡張子を別の方法でカバーしている他のプラグインのLSP機能には波及しません。`/plugin` のErrorsタブでどのプラグインのLSPサーバーが失敗しているかを確認し、該当バイナリを個別にインストールすれば解決できます。

```bash
# 例: rust-analyzerが未インストールの場合
rustup component add rust-analyzer
```

## 注意点

- LSPプラグインはあくまで「Claude Codeが言語サーバーとどう接続するか」を設定するものであり、言語サーバー本体（バイナリ）は別途インストールする必要があります。
- 同じ拡張子を複数のプラグインが扱う設定自体は引き続き可能ですが、意図せず重複させると診断結果が二重に表示されたり、想定外のLSPサーバーが優先されたりする可能性があるため、必要なLSPプラグインだけを有効化することが推奨されます。
- LSPサーバーの読み込み状況やエラーは `/plugin` の詳細ビューやErrorsタブ、`claude plugin details <name>` から確認できます。
- プロジェクトスコープのプラグインが提供するLSPサーバーは、ワークスペースを信頼するまで起動しません。
- プラグイン更新後もセッション内では旧バージョンのパスを使い続けるため、LSPサーバーを新しいバージョンに切り替えるには `/reload-plugins` を実行する必要があります。
