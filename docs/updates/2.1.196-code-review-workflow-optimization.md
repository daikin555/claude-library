## 原文（日本語に翻訳）

`/code-review` のワークフローを改善：5つあったクリーンアップ検出器を1つに統合し、トークン使用量を約25%削減

## 原文（英語）

Improved `/code-review` workflow: merged five cleanup finders into one, cutting token usage by roughly 25%

## 概要

`/code-review` コマンドの内部ワークフローが改善されました。これまでコードの再利用性・簡潔性・効率性などの「クリーンアップ」観点を検出するために5つの独立した検出器（finder）が個別に実行されていましたが、これらを1つに統合しました。結果として、同等のレビュー品質を保ちながら、トークン使用量を約25%削減することに成功しています。

## 基本的な使い方

```bash
# 現在の差分に対してコードレビューを実行
/code-review

# レビュー結果の提案を自動的にワーキングツリーへ適用
/code-review --fix

# GitHub PRにインラインコメントとして投稿
/code-review --comment
```

## 実践例

### 通常のコードレビュー実行

```bash
/code-review
# 出力例:
# - 正確性に関するバグの指摘
# - 再利用・簡潔化・効率化に関するクリーンアップ提案
# （クリーンアップ検出が1パスにまとまったことでトークン消費が削減）
```

### effort levelの指定

```bash
# 少数の高信頼度な指摘のみ欲しい場合
/code-review low

# デフォルトの中間レベル
/code-review medium

# より広いカバレッジで網羅的にレビューしたい場合
/code-review high
```

### トークン使用量削減の効果

```
【修正前】
クリーンアップ観点ごとに5つの検出器を個別実行
→ 同じコード差分に対して5回分のコンテキスト読み込みが発生

【修正後】
5つの観点を1つの検出器に統合して1回で実行
→ 同じコード差分に対するコンテキスト読み込みが1回で済み、
   トークン使用量が約25%削減
```

PRや差分が大きいほど、この削減効果は累積的に効いてきます。複数のPRを継続的にレビューする運用では、コスト削減につながります。

## 注意点

- この変更はコマンドの使い方自体には影響しません。既存のオプション（`--fix`、`--comment`、effort levelの指定など）はそのまま利用できます
- トークン削減の対象はクリーンアップ検出フェーズのみで、正確性バグの検出ロジックには変更はありません
- 検出器の統合によりレビュー品質が低下することはなく、これまでと同等の指摘が得られるよう設計されています
