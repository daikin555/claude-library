## 原文（日本語訳）

`claude agents`プロセスが起動していたバイナリが自動アップデートによって置き換えられた後、バックグラウンドセッションへのアタッチが「Couldn't start the background daemon」というエラーで恒久的に失敗する不具合を修正した。

## 原文（英語）

Fixed background-session attach failing permanently ("Couldn't start the background daemon") after an update replaced the binary a running `claude agents` process was launched from

## 概要

Claude Codeのバックグラウンドセッションは、per-userのスーパーバイザー（デーモン）プロセスによってホストされており、通常はインストール済みバイナリの変更をファイル監視で検知し、自動アップデート後に新しいバージョンへ自動的に再起動・再接続します。しかし特定の状況下では、実行中の`claude agents`プロセスが起動時点のバイナリを参照し続けたままアップデートでそのファイルが置き換えられてしまい、以後デーモンを起動できず「Couldn't start the background daemon」というエラーが恒久的に出続けるバグがありました。今回のアップデートでこの問題が修正されています。

## 基本的な使い方

通常通り`claude agents`でAgent Viewを開き、バックグラウンドセッションにアタッチするだけです。アップデートを挟んでも意識する必要はありません。

```text
claude agents
# セッションを選択して Enter または → でアタッチ
```

## 実践例

### CLIの自動アップデートをまたいだ長時間運用

`claude agents`を起動したまま長時間PCを使い続け、その間にCLIが自動アップデートされるケースでも、以前は再アタッチ時にデーモンが起動できなくなる恐れがありました。修正後は、アップデートを挟んでもスーパーバイザーが新しいバイナリで正しく再起動され、既存のバックグラウンドセッションへの接続が維持されます。

```text
claude agents
# （この間にCLIが自動アップデート）
claude attach <id>
# → 修正前: "Couldn't start the background daemon" が発生し続ける
# → 修正後: 正常にアタッチできる
```

### シェルから直接セッションIDを指定してアタッチする場合

Agent Viewを介さず`claude attach <id>`で直接アタッチする運用でも、同様にアップデート後の恒久的な失敗が解消されています。

```text
claude attach abc123
```

## 注意点

- スーパーバイザーはインストール済みバイナリをファイル監視しており、ネットワークチェックではなくローカルのファイル変更を契機に新バージョンへ再起動します。
- バックグラウンドセッション自体はデタッチされたプロセスとして動作しているため、スーパーバイザーの再起動中もセッションは動き続けます。
- アイドル状態のセッションは新バージョンへ段階的に移行され、作業中・入力待ち・アタッチ中のセッションは次回のプロセス再起動時に新バージョンへ移行されます。
